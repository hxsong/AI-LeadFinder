# AI-LeadFinder 静态线索深度挖掘与打分引擎 (P22-P27) 逻辑实现白皮书

> **版本**：v1.0.0
> **状态**：N（初稿）
> **关联原型页面**：线索详情.html
> **最后更新**：2026-04-10


本模块涵盖了系统**阶段二末端：精准联系人挖掘与线索就绪度 (Lead Readiness) 打分**的核心工程实现路径。此阶段的核心不仅是获取联系人，更是**完成数据从“临时搜索态”向“永久线索态”的跨越**。高分企业一旦被选中，其所有底层的静态证明材料将连同联系人一起被永久固化入库。

## 商业化底层：价值导向的动态后置计费模型 (Value-Based Post-Billing)

采用**“见效计费，多维溢价”**，让客户为真实的商业价值买单。

* **计费规则**：获取 1 个带有验证邮箱的联系人 = 2 Credits；附加 LinkedIn 主页 = +1 Credit；附加 Twitter/FB = +1 Credit/个。
* **保底策略**：单次无上限获取，后台精准算人头。若 API 仅返回空列表（未查到邮箱），则本次操作扣费为 0，但该企业的基础静态信息仍会免费入库保存。

## P22：选择目标企业挖掘联系人 (异步交互起点)

**业务目标**：完成前端交互与后置扣费契约确认，并启动数据持久化动作。

* **📥 输入参数**：
* `company_name` (必填，如 "Orlando Mega Stage AV")
* `company_domain` (选填，如 "orlando-av.com"。*解决前期静态抓取未匹配到官网的断层情况*)
* `full_static_payload` (必填，包含 P21 展示的匹配分、AI判词、海关/展会原始证明数据)。


* **⚙️ 异步交互流转 (极致 UX 设计)**：
1. 用户在 P21 列表点击 `[挖掘关键联系人]`。
2. 系统弹窗提示："*将按实际获取价值扣费（2 Credits/有效邮箱，社媒主页额外计费）。由于全网深度挖掘需要时间，任务将在后台异步执行。*"
3. **页面跳转**：用户点击确认后，**当前页面立即跳转至【静态线索列表页 (Static Lead Pool)】**。
4. **进度展示**：在对应的线索卡片下方显示进度条区域：
   - **挖掘中状态**：显示进度条和"正在深度挖掘决策人网络..."提示，卡片顶部显示"挖掘中"标签
   - **已完成状态**：进度条区域替换为联系人结果展示，显示已获取的联系人姓名和职位
   - 支持同时展示多个挖掘中的线索，后台进入 P23 异步执行流

## P24 - P25：API 调度与联系人数据获取 (单点突破架构)

**业务目标**：精准调度 Apollo 接口，兼容“有域名”与“无域名”双重场景。

* **🛠️ 核心执行链路**：
自动化执行引擎 (AEE) 调用 Apollo.io 的 `/v1/mixed_people/search` 接口。
* **参数路由逻辑**：
* `IF (company_domain 存在)`: 使用 `q_organization_domains={company_domain}` 进行精准锁定。
* `ELSE`: 使用 `q_organization_name={company_name}` 进行模糊匹配。


* **数据过滤**：筛选返回数据中 `email_status="verified"` 的实体记录。


* **📤 输出内容**：一组包含姓名、职位、邮箱及多维度社媒链接的原始数组 `raw_contacts_list`。（注：若未查到，则输出 `[]`，系统将流转至后续步骤进行 0 元入库）。

---

## P25：AI 结合数据进行线索匹配智能分析 (乘数打分引擎)

**业务目标**：大模型负责"选将排序"与"提权定级"，剔除无效干扰人员。

* **📥 输入内容**：P24 获取的 `raw_contacts_list` + P09 `ICP_LeadLevel` + P19 `account_fit_score` (如：96分)。
* **🛠️ 模型选型**：`gpt-4o-mini`。
* **⚙️ 核心 Prompt (Markdown 格式)**：

```markdown
# Role
你是一位资深的 B2B 销售总监与线索质量评估专家。

# Input Data
- 企业基础匹配分 (Base Score): {account_fit_score}
- 理想决策人画像 (ICP Lead Criteria): {icp_lead_criteria}
- 候选联系人列表: {raw_contacts_list}

# Task
1. **异常处理**：如果候选联系人列表为空 `[]`，直接输出 final_lead_score = Base Score，并说明“遗憾未找到关键人”。
2. **沙里淘金 (Ranking)**：根据联系人的职位，评估其采购决策权。保留核心业务相关联系人，并按重要程度排序（No.1 必须是最高决策者）。
3. **执行乘数提权 (Apply Multiplier)**：
   - 根据你排在 No.1 的【核心决策人】职位进行提权：
   - 决策层 (C-level, VP, Director): 系数 1.1
   - 执行层 (Manager, Engineer): 系数 1.0
   - 普通级 (Info, Sales): 系数 0.8
   - 计算公式：最终线索分 = Base Score * 系数 (四舍五入取整)。
4. **生成 AI 判词**：结合原有的企业优势，向客户解释这名关键决策人的价值。

# Output JSON Schema
{
  "top_contacts_array": [
    {
      "name": "提取的名字",
      "title": "提取的职位",
      "email": "提取的邮箱",
      "social_links": {"linkedin": "url", "twitter": "url"}
    }
  ],
  "scoring_result": {
    "multiplier_applied": 1.1,
    "final_lead_score": 105
  },
  "ai_reasoning": "企业基础盘吻合。现已为您找到采购副总裁...完全符合决策人画像，跃升为 105 分 S 级线索。"
}

```

## P26：执行后台精算与线索池持久化入库

**业务目标**：完成资金清算，并将所有零散数据合并，固化为永久资产。

* **⚙️ 后台组装与流转**：
1. **价值精算 (Billing Logic)**：
若 `top_contacts_array` 为空，计 0 分；否则遍历计算：`Cost = Sum (每人 2 基础分 + 领英 1 分 + 其他社媒 1 分)`。扣除积分并生成账单流水。
2. **数据融合与入库 (Persistence)**：
将 P22 传来的 `full_static_payload` (企业名、域名、海关/展会原始数据) 与 P25 生成的 `top_contacts_array`、`final_lead_score` 进行合并。
完整存储至数据库的 **静态线索表 (Static Leads Table)**。
