# AI-LeadFinder 静态数据搜客与打分引擎 (P12-P21) 逻辑实现白皮书

> **版本**：v1.0.0
> **状态**：N（初稿）
> **关联原型页面**：搜客需求输入.html、搜客策略预览.html、搜客执行中.html
> **关联功能说明文档**：../prototype/docs/搜客需求输入-说明.md、../prototype/docs/搜客策略预览-说明.md、../prototype/docs/搜客执行中-说明.md
> **最后更新**：2026-04-16


本模块涵盖了系统**阶段二：全网搜客、数据抓取与静态线索评估**的核心工程实现路径。引擎将严格继承阶段一动态生成的 ICP 画像基准，通过 AI 动态规划 API 调度路径，结合底层计费模型进行费效核算，并输出高转化率的企业线索列表。

## 商业化底层：基于 API 调用的动态计费换算模型

采用“AI 预估调用次数 -> 后台加权折算 -> 实际执行后多退少补”的计费闭环，确保 SaaS 商业模式不亏损。

1. **系统预设 API 单价表 (后台配置)**：
* `Maps_API`: 2 Credits / 次调用 (单次返回 20 条基础记录)
* `CHUHAIPAI_API`: 5 Credits / 次调用 (支持批量查询 100 个域名的海关)
* `10TIMES_EXPO_API`: 3 Credits / 次调用 (支持批量查询 100 个域名的展会)
* `JINA_READER_API` (文本抓取): 1 Credit / 次调用 (单域名解析)


2. **预估计费换算公式 (在 P13 节点后执行)**：
* `API 预估基准成本` = ∑ (每种 API 预计调用次数 × 对应 API 单价)
* `总预估消耗 Credits` = API 预估基准成本 × (1 + 动态溢价比例)
* *注：动态溢价比例（如 30%）用于覆盖难以精准计算的大模型 Token 消耗、并发服务器开销以及保障系统基础毛利。*


3. **实际扣费机制**：在 P21 节点检索结束后，系统按真实的 API 成功调用次数重新核算，从用户余额中扣除实际点数 (Credits)。

## P12：输入自然语言描述补充搜客需求

**业务目标**：接收企业管理员 (EBA) 当下的短期战术搜索意图。

* **📥 输入参数**：`user_query` (String) - 用户的自然语言指令（如：“找一下加州的音视频集成商”）。
* **⚙️ 降级兜底 (Fallback)**：若 `user_query` 为空，前端隐式传递 `use_default_icp=true` 标识，系统将完全调取 [P09] 阶段的 ICP 画像作为默认搜索指令。

## P13：结合业务分析结果 AI 智能生成搜客策略

**业务目标**：大模型充当“API 规划师”，将需求转化为三方接口调度树，并预估每种 API 的调用次数。

* **📥 输入内容**：P12 的 `user_query` + P09 的 `ICP 建模 JSON` (包含维度与动态权重)。
* **🛠️ 模型选型**：`gpt-4o` 或 `Claude 3.5 Sonnet`。
* **⚙️ 核心 Prompt (Markdown 格式)**：

```markdown
# Role
你是一个智能 API 调度网关与搜客策略规划师。

# Input
- 短期搜索意图：{user_query} （若为空，请完全依赖底层画像）
- 底层 ICP 画像基准：{icp_base_profile}

# Task
请将上述意图融合，制定一套瀑布流式的数据抓取策略。你必须决定：先调用哪个 API 铺网？再调用哪个 API 过滤？并合理预估所需的 API 调用次数。

# Available APIs (可用数据源)
1. `Maps_API`: 擅长按地理位置(`location`)和实体类型寻找企业，单次调用返回 20 条。
2. `CHUHAIPAI_API`: 擅长寻找有跨国采购记录的买家，支持单次 100 个域名的批量查询。
3. `10TIMES_EXPO_API`: 擅长寻找近期活跃参展商，支持单次 100 个域名的批量查询。
4. `JINA_READER_API`: 网页纯文本解析，1 次调用对应 1 个域名。

# Execution Rules
1. 目标寻客数量上限设为 {target_count} 家。
2. **动态起手式**：
   - 意图包含明确区域，优先用 `Maps_API` 起手。
   - 意图强调采购某类产品，优先用 `CHUHAIPAI_API` 起手。

# Output JSON Schema
{
  "search_strategy_rationale": "用一句话向用户解释这套策略的逻辑",
  "estimated_api_calls": {
    "GOOGLE_MAPS_API": 5, // 例如：找 100 家需调用 5 次
    "CHUHAIPAI_API": 1,
    "10TIMES_EXPO_API": 1,
    "JINA_READER_API": 100
  },
  "waterfall_execution_plan": [
    {
      "step": 1,
      "api_source": "GOOGLE_MAPS_API",
      "parameters": {"query": "AV Integrators in California", "type": "electronics_store", "limit": 100}
    },
    {
      "step": 2,
      "api_source": "CHUHAIPAI_API",
      "action": "bulk_verify_step_1_results",
      "parameters": {"hs_code": "852859", "timeframe": "last_12_months"}
    }
  ]
}

```

## P14 - P16：展示搜客策略和预估消耗费用并确认

**业务目标**：后台介入计算费用，前端透明化展示策略，阻断成本失控风险。

* **后台运算逻辑**：CLS 系统接收 P13 生成的 JSON，读取 `estimated_api_calls`，套用上述计费公式（含 30% 溢价），算出 `estimated_credits`。
* **交互流转 (P15-P16)**：
* **UI 提示文案**：“AI 搜客规划已就绪。系统将优先在加州地图雷达扫描目标实体，并结合海关与展会数据进行深度交叉信调。本次深度检索预估最多消耗 **148 Credits**（实际扣费将按真实检索成功量结算）。是否确认执行？”
* **动作**：用户点击“确认开始搜客” (P16)，系统冻结对应预估点数，正式进入异步抓取流。

## P17 - P18：发送策略与接口调度抓取 (API Chaining)

**业务目标**：SDA 根据搜客策略逐步调用 TDS，并利用文本解析 API 填补非结构化文本的空白。

* **⚙️ 核心执行链路**：
1. **核心数据拉取**：按计划请求 Google Maps / 展会 API，获取基础 `Raw Data List`（含企业名、URL、地理位置）。
2. **批量交叉验证 (Bulk Match)**：将名单组装成数组，一次性发给出海派和 10times API，标记出 `has_customs` 和 `has_exhibition`。
3. **文本富化 (打分前置条件)**：
* **有 URL 的**：并发调用 `Jina Reader API`，抓取官网前 800 字正文。
* **无 URL 的**：通过低成本搜索引擎接口获取摘要 (Snippet)。




* **📤 输出内容**：组装完成的异构数据字典数组。

## P19：四维评分计算与匹配度分析

**业务目标**：严格继承动态 ICP 模型权重，综合四维评分框架，输出最具商业价值的 **企业匹配分 (Account Fit Score)**。

* **📥 输入内容**：P18 富化后的异构数据包 + P09 的动态 `ICP_Scoring_Model` 与 `业务类型权重配置`。

* **⚙️ 四维评分框架 (满分 100分)**：

引擎实时解析 P09 设定的动态权重（根据业务类型动态生成）。

**维度一：需求匹配度（AI 模型计算）**

使用 AI 模型判断候选企业是否是客户的潜在客户：
* 输入客户信息和候选企业信息（统一维度框架）
* AI 分析供需关系、品类匹配、业务模式匹配、市场范围匹配
* 输出 0-100 分的需求匹配度分数
* 推荐模型：`gpt-4o-mini`（批量计算）+ `gpt-4o`（高分二次验证）

**维度二：购买能力（规模 + 地理）**

```
购买能力得分 = (规模得分 × 0.6) + (地理得分 × 0.4)

规模评分（满分 60 分）：
├── 大型企业（500人以上）：60 分
├── 中型企业（50-500人）：45 分
├── 小型企业（10-50人）：30 分
├── 微型企业（10人以下）：15 分
└── 无数据：30 分（默认中等偏下）

地理评分（满分 40 分）：
├── 在目标区域内：40 分
├── 不在目标区域内：0 分
└── 无数据：20 分（默认中等）
```

**维度三：采购意向（海关记录 + 时间衰减）**

```
采购意向得分 = HS Code 匹配得分 × 时间衰减系数

HS Code 匹配规则：
├── 有对口 HS Code 进口记录（前4位相同）：100%
├── 有相关 HS Code 进口记录（前2位相同）：70%
├── 有进口记录但 HS Code 不匹配：40%
└── 无海关记录：0%

时间衰减系数：
├── 0-30天：系数 = 1.0（不衰减）
├── 31-90天：系数 = 0.95
├── 91-180天：系数 = 0.85
├── 181-365天：系数 = 0.70
└── 超过365天：系数 = 0.50
```

**维度四：活跃意向（展会记录 + 时间衰减）**

```
活跃意向得分 = 展会匹配得分 × 时间衰减系数

展会匹配规则：
├── 参展商身份参加行业展会：100%
├── 观众身份参加行业展会：60%
├── 参加非行业展会：30%
└── 无展会记录：0%

时间衰减系数：与采购意向相同
```

* **⚙️ 综合评分计算**：

```
潜在客户得分 = 需求匹配度 × W1 
             + 购买能力 × W2 
             + 采购意向 × W3 
             + 活跃意向 × W4

其中 W1 + W2 + W3 + W4 = 1.0
权重根据客户业务类型动态生成（见 P09）
```

* **📊 评分示例（外贸型企业）**：

```
权重配置：需求匹配度 32%、购买能力 20%、采购意向 35%、活跃意向 13%

维度得分：
├── 需求匹配度：88 分（AI 判断供需高度匹配）
├── 购买能力：43 分（中型企业 + 目标区域内）
├── 采购意向：67 分（命中2个 HS Code，30天内进口）
└── 活跃意向：68 分（参展商身份，120天前参展）

最终得分 = 88 × 0.32 + 43 × 0.20 + 67 × 0.35 + 68 × 0.13
         = 28.16 + 8.60 + 23.45 + 8.84
         = 69.05 分
```

* **生成最终判词 (Explainability)**：调用轻量级 LLM，将高分特征融合成 AI 推荐理由。
