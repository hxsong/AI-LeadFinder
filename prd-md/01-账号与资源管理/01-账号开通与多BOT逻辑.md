# AI-LeadFinder 账号初始化与多 BOT 工作空间 (P01-P03) 逻辑实现白皮书

> **版本**：v1.0.0
> **状态**：N（初稿）
> **关联原型页面**：多BOT工作台.html
> **最后更新**：2026-04-10


本模块涵盖了系统**阶段一：服务开通、租户初始化与多 BOT 业务建模**的前置核心工程路径。系统确立了 **“账号级统管算力资金池 (Global Credits) + BOT 级绝对隔离业务数据 (Workspace)”** 的底层架构，以支撑多品牌、多产品线的企业级客户场景。

## 商业化底层：SYS 平台运营系统的新增 SKU 与资源定义

在已有的平台运营系统 (SYS) 中，无需开发全新的计费大模块，仅需在现有的“租户资源包管理”功能中，上架两种全新的基础资源类型 (SKU)。

* **资源类型 1：Global Credits (全局获客算力积分)**
* **属性**：消耗型资产 (Consumable)。
* **作用**：全账号名下所有 BOT 共享此资金池，用于支付全网搜客、IP 解析、联系人挖掘等动作的 API 与 Token 成本。


* **资源类型 2：BOT Seats (获客引擎实例配额)**
* **属性**：并发限制型资产 (Concurrency Limit)。
* **作用**：限制该企业账号最多可同时创建并运行几个独立的获客 BOT（不同 BOT 对应不同的独立站或业务线）。

## P01 - P02：审核订单、开通账号与注入全局资源

**业务目标**：平台管理员 (SAD) 在后台完成线下销售订单的履约，为客户开启 SaaS 租户权限，并下发初始资源包。

* **🛠️ SYS 运营系统后台样式示例 (逻辑说明，非新UI)**：
在 SYS 现有的【客户/租户管理 -> 资源充值】面板中，操作界面如下：
> **目标租户**: `Orlando Mega Stage AV (Tenant_ID: T-8821)`
> **操作类型**: [充值/修改资源包 ▾]
> **新增资源包 SKU**:
> ☑️ `[LeadFinder 旗舰版基础包]`：包含 3 个 BOT 席位 + 50,000 Global Credits。
> ☑️ `[LeadFinder 算力加油包]`：包含额外 10,000 Global Credits。
> **操作**: `[确认下发]`


* **⚙️ 系统间流转 (API 同步)**：
1. SYS 系统生成充值流水记录。
2. SYS 调用数据获客主系统 (CLS) 的内部 Open API：`POST /internal/v1/tenant/resources/provision`。
3. CLS 接收指令，初始化该企业租户的数据库表，将 `total_credits` 更新为 60,000，将 `max_bot_seats` 更新为 3。

## 终局视角：P03 登录工作台与多 BOT 聚合面板

**业务目标**：企业管理员 (EBA) 登录后的首个“指挥中心 (Command Center)”。向客户清晰展示剩余资产，并提供进入或创建隔离业务空间（BOT）的入口。

* **UI 呈现核心元素**：
* **全局资源看版 (Global Resource Dashboard)**：
* 💰 **算力积分余额**：`60,000 Credits`
* 🤖 **引擎实例额度**：`已创建 1 / 共 3 个 BOT`
* **行动召唤 (CTA)**：`[购买更多算力/扩展 BOT 席位]`按钮（注意：点击不直接跳支付，而是弹窗）。


* **BOT 工作空间矩阵 (Workspace Matrix)**：
* **已激活的 BOT 卡片**：展示 BOT 名称（如“北美区-户外屏主站”）、当前运行状态、该 BOT 累计获取的线索数。卡片上提供主按钮 `[进入获客引擎]`。
* **创建新 BOT 占位符**：一个带有 `+` 号的虚线框卡片。若剩余额度 > 0，则允许点击创建；若额度不足，则置灰并提示联系销售。

## P03：创建/购买并进入特定的「获客服务 BOT」实例

**业务目标**：完成前端界面渲染，实现业务隔离空间的切换，并处理线下销售线索的收集。

* **📥 核心输出 JSON (驱动 P03 聚合面板渲染)**：

```json
{
  "tenant_id": "T-8821",
  "tenant_name": "Orlando Mega Stage AV",
  "global_resources": {
    "credits_balance": 60000,
    "bot_seats_total": 3,
    "bot_seats_used": 1,
    "bot_seats_available": 2
  },
  "bot_instances": [
    {
      "bot_id": "BOT-1001",
      "bot_name": "北美区-LED大屏主站",
      "status": "active",
      "created_at": "2026-02-25",
      "stats": {
        "static_leads_generated": 142,
        "dynamic_leads_captured": 38,
        "credits_consumed": 2150
      },
      "cta": "进入获客引擎"
    }
  ]
}

```

* **⚙️ 关键交互逻辑与防越权控制**：
**动作 1：点击 [购买更多算力/扩展 BOT 席位] (线下销售导流)**
* 系统弹出【联系专属客户成功经理】模态框。
* **UI 文案**：“*如需扩充获客引擎实例席位，或补充全局获客算力积分，请联系您的专属销售顾问为您配置专属升级方案。*”
* 展示内容：销售人员的姓名、企业微信二维码、联系电话。


**动作 2：点击 [+ 创建全新的获客服务 BOT]**
* **校验机制**：前端校验 `bot_seats_available > 0`。
* **业务流转**：若额度充足，弹出向导让用户输入“新 BOT 名称”并选择“语言区域”。确认后，系统在底层新建一个隔离的 `bot_id`，并立刻引导用户进入后续的 **[P04/P05] 业务材料上传与独立站授权流**，完成该新 BOT 的冷启动初始化。


**动作 3：点击现有 BOT 卡片上的 [进入获客引擎]**
* **上下文切换 (Context Switch)**：前端将全局状态中的 `current_active_bot_id` 设置为选中的 ID（如 `BOT-1001`）。
* 页面路由跳转至该 BOT 专属的分析、搜客与线索列表页。此后该用户进行的所有需求匹配度计算、静态规则打分、探针埋点，**均被严格死锁在该 `BOT-1001` 的数据作用域内**，绝不会与其他站点的配置发生交叉污染。
