# 02-modules 模块层设计规范

本目录存放各功能模块的详细设计文档，承接概念层的业务流设计，将业务流程拆解为可独立开发的功能模块。

---

## 职责定位

| 维度 | 说明 |
|------|------|
| **核心问题** | 业务怎么运作？用户怎么用？ |
| **关注点** | 业务规则、用户故事、验收标准 |
| **描述方式** | 用户视角、业务视角 |
| **读者对象** | 产品经理、设计师 |
| **变更频率** | 中（随业务调整） |

---

## 文件结构

```
02-modules/
├── 01-{模块A}.md
├── 02-{模块B}.md
├── 03-{模块C}.md
└── README.md
```

---

## 编写原则

1. **用户视角**：从用户角度描述功能，不涉及技术实现细节
2. **故事驱动**：以用户故事为核心，描述用户如何使用功能
3. **规则明确**：业务规则清晰、可执行、无歧义
4. **验收标准**：每个功能都有明确的验收条件

## 章节编写规范

### 功能目标

**定义**：本模块要解决什么业务问题，带来什么价值。

**格式**：
```markdown
## 功能目标

本模块涵盖{业务范围}的全链路功能。核心目标包括：

1. **{目标1}**：{具体描述}
2. **{目标2}**：{具体描述}
...
```

### 用户故事

**定义**：从用户角度描述功能使用场景，采用"作为...我想要...以便..."格式。

**格式**：
```markdown
## 用户故事

### 故事一：{故事名称}

**作为** {用户角色}
**我想要** {执行的操作}
**以便于** {达到的目的}

**验收条件**：
- 条件1
- 条件2
```

### 业务规则

**定义**：业务层面的规则，描述"业务如何运作"，而非"技术如何实现"。

**格式**：
```markdown
## 业务规则

### 规则一：{规则名称}

- {规则内容1}
- {规则内容2}
...

### 规则二：{规则名称}

- {规则内容1}
- {规则内容2}
```

### 用户操作流程

**定义**：从用户角度描述操作步骤，不涉及系统内部处理。

**格式**：
```markdown
## 用户操作流程

### 场景一：{场景名称}

1. 用户{操作1}
2. 用户{操作2}
3. 系统显示{结果}
...

### 场景二：{场景名称}

1. 用户{操作1}
2. 系统显示{结果}
...
```

### 界面元素

**定义**：页面包含的界面元素及其说明，为原型设计提供依据。

**格式**：
```markdown
## 界面元素

### {页面名称}

| 元素 | 类型 | 说明 | 必填 |
|------|------|------|------|
| {元素名} | {类型} | {说明} | {是/否} |
```

### 验收标准

**定义**：功能完成的验收条件，用于判断功能是否开发完成。

**格式**：
```markdown
## 验收标准

### 功能验收

- [ ] {验收条件1}
- [ ] {验收条件2}
...

### 体验验收

- [ ] {体验条件1}
- [ ] {体验条件2}
...
```

## 模块拆分原则

### 拆分依据

模块划分应与概念层的业务流设计保持一致：

```
01-concept/05-业务流设计.md
    ↓ 指导模块拆分
02-modules/
    ├── 01-{基础模块}.md      # 前置基础模块
    ├── 02-{核心模块}.md      # 核心功能模块（可并行）
    ├── 03-{核心模块}.md
    └── 04-{支撑模块}.md      # 运营支撑模块
```

### 拆分规则

1. **阶段对应**：一个业务阶段对应一个或多个模块
2. **功能完整**：每个模块包含完整的功能闭环
3. **边界清晰**：模块间边界明确，避免功能重叠
4. **依赖明确**：模块间的依赖关系清晰
