# AI-LeadFinder 账号初始化与多 BOT 工作空间 (P01-P03) 逻辑实现白皮书

本模块涵盖了系统**阶段一：服务开通、租户初始化与多 BOT 业务建模**的前置核心工程路径。系统确立了 **“账号级统管算力资金池 (Global Credits) + BOT 级绝对隔离业务数据 (Workspace)”** 的底层架构，以支撑多品牌、多产品线的企业级客户场景。

## 商业化底层：SYS 平台运营系统的新增 SKU 与资源定义

在已有的平台运营系统 (SYS) 中，无需开发全新的计费大模块，仅需在现有的“租户资源包管理”功能中，上架两种全新的基础资源类型 (SKU)。

* **资源类型 1：Global Credits (全局获客算力积分)**
* **属性**：消耗型资产 (Consumable)。
* **作用**：全账号名下所有 BOT 共享此资金池，用于支付全网搜客、IP 解析、联系人挖掘等动作的 API 与 Token 成本。


* **资源类型 2：BOT Seats (获客引擎实例配额)**
* **属性**：并发限制型资产 (Concurrency Limit)。
* **作用**：限制该企业账号最多可同时创建并运行几个独立的获客 BOT（不同 BOT 对应不同的独立站或业务线）。

## P01 - P02：审核订单、开通账号与注入全局资源

**业务目标**：平台管理员 (SAD) 在后台完成线下销售订单的履约，为客户开启 SaaS 租户权限，并下发初始资源包。

* **🛠️ SYS 运营系统后台样式示例 (逻辑说明，非新UI)**：
在 SYS 现有的【客户/租户管理 -> 资源充值】面板中，操作界面如下：
> **目标租户**: `Orlando Mega Stage AV (Tenant_ID: T-8821)`
> **操作类型**: [充值/修改资源包 ▾]
> **新增资源包 SKU**:
> ☑️ `[LeadFinder 旗舰版基础包]`：包含 3 个 BOT 席位 + 50,000 Global Credits。
> ☑️ `[LeadFinder 算力加油包]`：包含额外 10,000 Global Credits。
> **操作**: `[确认下发]`


* **⚙️ 系统间流转 (API 同步)**：
1. SYS 系统生成充值流水记录。
2. SYS 调用数据获客主系统 (CLS) 的内部 Open API：`POST /internal/v1/tenant/resources/provision`。
3. CLS 接收指令，初始化该企业租户的数据库表，将 `total_credits` 更新为 60,000，将 `max_bot_seats` 更新为 3。

* **UI 呈现核心元素**：
* **全局资源看版 (Global Resource Dashboard)**：
* 💰 **算力积分余额**：`60,000 Credits`

* **BOT 工作空间矩阵 (Workspace Matrix)**：

# AI-LeadFinder 业务初始化与分析流 (P04-P11) 逻辑实现白皮书

本模块涵盖了系统**阶段一：服务开通、租户初始化与业务建模** 的核心工程实现路径。重点解决了异构数据的接收、清洗、降维推理以及最终高维企业画像的生成。

## P04：自助上传企业/产品业务材料

**业务目标**：接收用户本地非结构化业务凭证，作为 AI 画像分析的物理语料。此流程与 P05/P06 互斥，防止实体信息冲突。

* **📥 输入内容/参数**：
* `file_streams`: 支持 PDF（产品画册）、PPTX（公司介绍）、JPG/PNG（资质证书）。单文件上限 20MB。
* `competitor_urls` (可选): 竞品独立站链接，用于反向推演行业特征。


* **🛠️ 工具与技术栈**：
* 存储层：AWS S3 / 阿里云 OSS。
* 解析层：LlamaParse / Unstructured.io（处理复杂排版 PDF 提取 Markdown），Tesseract OCR（处理纯图片）。


* **⚙️ 核心模型/Prompt**：无（纯工程物理转换节点）。
* **📤 输出内容**：带有大量排版噪音（如页眉、页脚、无意义数字）的 Markdown 长文本集合。

---

## P05 & P06：独立站授权与自动同步数据

**业务目标**：利用“独立站”底层优势，通过 API 毫秒级获取高纯度、高结构化的客户商业意图数据。

* **📥 输入内容/参数**：
* `tenant_id`: 租户唯一标识。
* `selected_site_id`: 用户在前端指定的单一核心主站 ID。


* **🛠️ 工具与技术栈**：建站系统 (WBS) 内部只读 API 网关。
* **⚙️ 核心模型/Prompt**：无。
* **📤 输出内容 (JSON Payload)**：
```json
{
  "seo_metadata": {"title": "...", "keywords": ["..."], "h1": ["..."]},
  "product_category_tree": {"level_1": ["..."], "level_2": ["..."]},
  "about_us_content": "...",
  "application_cases": ["..."]
}

```



---

## P07：提交全量材料进行 AI 智能业务分析

**业务目标**：数据获客主系统 (CLS) 向企业分析服务 (EAI) 提交处理请求，建立异步任务队列。

* **📥 输入内容/参数**：P04 的 Markdown 文本，或 P06 的结构化 JSON。
* **🛠️ 工具与技术栈**：Kafka / RabbitMQ 消息队列（处理大模型长耗时任务，实现接口削峰）。
* **📤 输出内容**：返回 `task_id`，前端开始轮询或通过 WebSocket 接收后续处理进度。

---

## P07.5：前置 OCR 文本提取

**业务目标**：对用户上传的 PDF、图片等非结构化文件进行 OCR 识别，提取原始文本内容，为后续 ETL 处理做准备。

* **🛠️ 工具与技术栈**：
  * **PDF 解析**：LlamaParse / Unstructured.io（处理复杂排版 PDF，提取 Markdown）
  * **图片 OCR**：Tesseract OCR / PaddleOCR（处理扫描件、证书图片）
  * **存储层**：AWS S3 / 阿里云 OSS

* **⚙️ 处理流程**：
  1. 判断文件类型（PDF/PPTX/图片）
  2. PDF/PPTX：调用 LlamaParse 提取 Markdown 文本
  3. 图片：调用 OCR 引擎识别文字
  4. 合并多文件提取结果

* **📤 输出内容**：带有排版噪音的原始 Markdown 文本集合，传递给 P08 进行结构化提取。

## P08：阶段 1 - 文本内容提取与结构化 (ETL)

**业务目标**：使用低成本、高并发的模型对 P04 的脏数据进行"脱水"和"提纯"，消除幻觉干扰，输出标准化的事实字典。

* **🛠️ 工具与模型**：`gpt-4o-mini` 或 `Gemini 1.5 Flash` (强制开启 JSON Mode)。
* **⚙️ 核心 Prompt (Markdown 格式)**：

```markdown
# Role
你是一个严谨的企业数据清洗与结构化抽取引擎 (ETL Data Extractor)。

# Task
请阅读以下可能含有大量排版噪音（如 PDF 页眉页脚、OCR 乱码）的文本片段，严格提取其中的客观业务事实，并按规定 JSON Schema 输出。

# Strict Rules
1. **零捏造原则**：绝对不要凭借自身先验知识捏造文档中未明确提及的信息。
2. **空值处理**：如果文档中未提及某类信息，对应字段必须返回 `null` 或空数组 `[]`。
3. **去营销化**：剥离所有主观形容词（如"世界领先"、"性价比之王"、"极致体验"），仅保留物理规格、标准术语、专有名词、资质认证。

# Expected JSON Output Schema
{
  "raw_company_profile": "提取企业定位、核心业务描述",
  "raw_company_history": "提取成立时间、工厂面积、员工数等客观数字",
  "raw_products_and_specs": ["提取具体产品型号及对应的核心物理/技术参数"],
  "raw_certifications": ["提取明确的 ISO/CE/FCC/专利等标准化证书名称"],
  "raw_core_capabilities": ["提取核心服务能力或技术优势"],
  "raw_target_markets": ["提取目标市场区域或国家"],
  "raw_target_customers": ["提取目标客户类型描述"]
}

```

* **📤 输出内容**：干净的实体字典 JSON，去除了一切排版噪音和冗余废话。

---

## P09：阶段 2 - 深度业务分析与 ICP 建模

**业务目标**：企业分析服务 (EAI) 的核心"大脑"。对提纯后的数据进行高维商业推演，生成覆盖五大维度的全景报告，并建立四维评分模型。

* **🛠️ 工具与模型**：
* **推理模型**：`Claude 3.5 Sonnet` 或 `gpt-4o` (具备极强的 B2B 产业链逻辑推演能力)。


* **⚙️ 核心 Prompt (Markdown 格式)**：

```markdown
# Role
你是一位拥有 15 年经验的顶尖外贸 B2B 商业分析师与产品架构师（精通 APPEALS 框架与全球供应链法则）。

# Task
请综合分析输入的结构化企业事实数据，深度剖析其商业模式，并严格按照要求输出一份极具专业度的《企业级业务与市场洞察报告》JSON。

# Analytical Guidelines (分析准则)
1. **企业介绍提取**：高度凝练企业的核心生态位。
2. **主营产品梳理**：整合产品线，保留最核心的 SKUs 及其关键技术壁垒。
3. **行业供应链推演**：
   - 上游：该企业依赖的核心原材料或技术供应商。
   - 下游（核心发力点）：**突破直接买家认知**！基于产品特性，推演至少 3 类具有集成、分销或特定场景落地能力的高潜力/隐性 B2B 客群。
4. **竞争优势提炼**：提取 5-8 个极具行业辨识度的标签（如特有材质、认证、特殊参数）。**这些标签将用于 AI 模型判断供需关系，必须是具体的实体词，严禁使用泛化形容词。**
5. **ICP (理想客户画像) 建模**：
   - 建立 4 个维度的打分模型：需求匹配度、购买能力、采购意向、活跃意向。
   - 权重根据业务类型动态生成，总和为 1.0。
   - 业务类型识别：外贸型、生产型、服务型、渠道型。
   - 各维度打分逻辑：
     - **需求匹配度**：AI 模型判断候选企业是否是客户的潜在客户（供需关系分析）。
     - **购买能力**：企业规模评分（60%）+ 地理范围符合度（40%）。
     - **采购意向**：海关足迹数据，根据 HS Code 匹配度和时间衰减系数打分。
     - **活跃意向**：展会足迹数据，根据参展记录和时间衰减系数打分。

根据识别的业务类型，按以下规则分配权重：

| 业务类型 | 需求匹配度 | 购买能力 | 采购意向 | 活跃意向 |
|----------|------------|----------|----------|----------|
| 外贸型 | 32% | 20% | 35% | 13% |
| 生产型 | 40% | 25% | 20% | 15% |
| 服务型 | 35% | 25% | 15% | 25% |
| 渠道型 | 35% | 20% | 30% | 15% |

# Output JSON Format (必须严格遵循，不包含任何外部文本)
{
  "企业介绍": {
    "定位概述": "一句话说明企业是谁，解决什么问题",
    "核心背书": ["资质A", "规模B", "成立年份C"]
  },
  "主营产品清单": [

    "下游高潜客群": ["如：视听设备集成商", "如：大型舞台租赁公司", "如：商业地产开发商"]
  },
  "竞争优势": {
    "说明": "以下标签将用于 AI 模型判断供需关系",

  },
  "业务类型识别": "外贸型",
  "ICP建模": [
    {
      "维度": "需求匹配度",
      "打分逻辑": "AI 模型判断供需关系，分析候选企业是否是客户的潜在客户",
      "权重占比": 0.32,
      "权重说明": "根据业务类型动态生成"
    },
    {
      "维度": "购买能力",
      "打分逻辑": "企业规模评分（60%）+ 地理范围符合度（40%）",
      "权重占比": 0.20,
      "权重说明": "根据业务类型动态生成"

    {
```

* **📤 输出内容**：
1. **动作拦截/后端存储**：系统解析出 JSON 中的 `竞争优势.core_tags` 字段，存储核心标签，用于 AI 模型需求匹配度计算。

## P11：展示业务分析报告

**业务目标**：将深度的底层推理逻辑，转化为客户可理解、可感知的“Aha Moment”（顿悟时刻），建立客户对 SaaS 产品的极高信任度，并引导进入搜客阶段。

* **📥 输入内容**：P10 返回的覆盖五个维度的 JSON 报告。
* **🛠️ 工具与技术栈**：前端渲染框架 (React/Vue) + UI 组件库。
* **⚙️ 页面展现逻辑**：
* 将 `企业介绍` 和 `主营产品清单` 渲染为左侧的“自身能力”板块。
* 将 `行业供应链` 渲染为一个可视化的上下游流程图。
* 将 `竞争优势` 标签高亮显示，并带有 Tooltip 提示："*这些标签将用于 AI 模型判断供需关系*"。

# AI-LeadFinder 静态数据搜客与打分引擎 (P12-P21) 逻辑实现白皮书

本模块涵盖了系统**阶段二：全网搜客、数据抓取与静态线索评估**的核心工程实现路径。引擎将严格继承阶段一动态生成的 ICP 画像基准，通过 AI 动态规划 API 调度路径，结合底层计费模型进行费效核算，并输出高转化率的企业线索列表。

## 商业化底层：基于 API 调用的动态计费换算模型

采用“AI 预估调用次数 -> 后台加权折算 -> 实际执行后多退少补”的计费闭环，确保 SaaS 商业模式不亏损。

1. **系统预设 API 单价表 (后台配置)**：
* `Maps_API`: 2 Credits / 次调用 (单次返回 20 条基础记录)
* `CHUHAIPAI_API`: 5 Credits / 次调用 (支持批量查询 100 个域名的海关)
* `10TIMES_EXPO_API`: 3 Credits / 次调用 (支持批量查询 100 个域名的展会)
* `JINA_READER_API` (文本抓取): 1 Credit / 次调用 (单域名解析)


2. **预估计费换算公式 (在 P13 节点后执行)**：
* `API 预估基准成本` = ∑ (每种 API 预计调用次数 × 对应 API 单价)
* `总预估消耗 Credits` = API 预估基准成本 × (1 + 动态溢价比例)
* *注：动态溢价比例（如 30%）用于覆盖难以精准计算的大模型 Token 消耗、并发服务器开销以及保障系统基础毛利。*


3. **实际扣费机制**：在 P21 节点检索结束后，系统按真实的 API 成功调用次数重新核算，从用户余额中扣除实际点数 (Credits)。

## P19：四维评分计算与匹配度分析

**业务目标**：严格继承动态 ICP 模型权重，综合四维评分框架，输出最具商业价值的 **企业匹配分 (Account Fit Score)**。

* **📥 输入内容**：P18 富化后的异构数据包 + P09 的动态 `ICP_Scoring_Model` 与 `业务类型权重配置`。

* **⚙️ 四维评分框架 (满分 100分)**：

引擎实时解析 P09 设定的动态权重（根据业务类型动态生成）。

**维度一：需求匹配度（AI 模型计算）**

使用 AI 模型判断候选企业是否是客户的潜在客户：
* 输入客户信息和候选企业信息（统一维度框架）
* AI 分析供需关系、品类匹配、业务模式匹配、市场范围匹配
* 输出 0-100 分的需求匹配度分数
* 推荐模型：`gpt-4o-mini`（批量计算）+ `gpt-4o`（高分二次验证）

**维度二：购买能力（规模 + 地理）**

```
购买能力得分 = (规模得分 × 0.6) + (地理得分 × 0.4)

规模评分（满分 60 分）：
├── 大型企业（500人以上）：60 分
├── 中型企业（50-500人）：45 分
├── 小型企业（10-50人）：30 分
├── 微型企业（10人以下）：15 分
└── 无数据：30 分（默认中等偏下）

地理评分（满分 40 分）：
├── 在目标区域内：40 分
├── 不在目标区域内：0 分
└── 无数据：20 分（默认中等）
```

**维度三：采购意向（海关记录 + 时间衰减）**

```
采购意向得分 = HS Code 匹配得分 × 时间衰减系数

HS Code 匹配规则：
├── 有对口 HS Code 进口记录（前4位相同）：100%
├── 有相关 HS Code 进口记录（前2位相同）：70%
├── 有进口记录但 HS Code 不匹配：40%
└── 无海关记录：0%

时间衰减系数：
├── 0-30天：系数 = 1.0（不衰减）
├── 31-90天：系数 = 0.95
├── 91-180天：系数 = 0.85
├── 181-365天：系数 = 0.70
└── 超过365天：系数 = 0.50
```

**维度四：活跃意向（展会记录 + 时间衰减）**

```
活跃意向得分 = 展会匹配得分 × 时间衰减系数

展会匹配规则：
├── 参展商身份参加行业展会：100%
├── 观众身份参加行业展会：60%
├── 参加非行业展会：30%
└── 无展会记录：0%

时间衰减系数：与采购意向相同
```

* **⚙️ 综合评分计算**：

```
潜在客户得分 = 需求匹配度 × W1 
             + 购买能力 × W2 
             + 采购意向 × W3 
             + 活跃意向 × W4

其中 W1 + W2 + W3 + W4 = 1.0
权重根据客户业务类型动态生成（见 P09）
```

* **📊 评分示例（外贸型企业）**：

```
权重配置：需求匹配度 32%、购买能力 20%、采购意向 35%、活跃意向 13%

维度得分：
├── 需求匹配度：88 分（AI 判断供需高度匹配）
├── 购买能力：43 分（中型企业 + 目标区域内）
├── 采购意向：67 分（命中2个 HS Code，30天内进口）
└── 活跃意向：68 分（参展商身份，120天前参展）

最终得分 = 88 × 0.32 + 43 × 0.20 + 67 × 0.35 + 68 × 0.13
         = 28.16 + 8.60 + 23.45 + 8.84
         = 69.05 分
```

* **生成最终判词 (Explainability)**：调用轻量级 LLM，将高分特征融合成 AI 推荐理由。

## P28 - P32：探针代码分发、授权与验证部署

**业务目标**：通过高度自动化的流程，降低企业管理员 (EBA) 在独立站 (WBS) 部署跟踪代码的技术门槛，并确保数据采集链路的连通性验证。

### 🔐 核心约束规则：域名绑定机制

**设计原则**：探针代码的生成与绑定以「网站域名」为最小单位，而非企业账号。

| 约束项 | 规则说明 | 业务理由 |
| --- | --- | --- |
| **一账号一域名** | 每个企业账号仅能绑定一个网站域名，不可同时绑定多个站点。 | 防止多站点访客数据重叠，避免后续数据合并时出现归因混乱。 |
| **探针域名唯一** | 探针代码针对域名生成，同一域名对应唯一探针 ID。 | 确保数据采集的准确性与可追溯性。 |
| **部署域名校验** | 探针部署后，系统验证部署域名是否与绑定域名一致。 | 防止探针代码被滥用至未授权域名。 |
| **域名变更限制** | 域名一旦绑定并生成探针，如需更换域名需联系平台管理员重置。 | 避免历史数据断层，保障数据连续性。 |

### ⚙️ 核心执行链路 (双分支部署架构)

**前置条件**：EBA 需先完成域名绑定，方可获取探针代码。

**域名绑定入口 (二选一)**：
* **方式 A - 独立站授权绑定**：通过 OAuth 授权独立站，系统自动提取域名并绑定。
* **方式 B - 手动填写域名**：EBA 手动输入网站域名，系统验证域名有效性后绑定。

**分支 A：已完成独立站授权 (自动化部署路径)**
* **[P28.1]** 系统检测到已授权独立站，自动提取域名并完成绑定。
* **[P29]** 数据获客主系统 (CLS) 提供一键安装按钮，准备自动部署探针。
* **[P30]** EBA 在前端执行一键安装触发。
* **[P31]** WBS 确认探针部署成功，系统校验部署域名与绑定域名一致性，向 CLS 回传站点验证状态。
* **[P32]** CLS 界面更新，向 EBA 显示「探针已生效，等待数据回传」。

**分支 B：手动填写域名 (手动埋点路径)**
* **[P28.1]** EBA 手动输入网站域名，系统验证域名格式与可访问性后完成绑定。
* **[P28.2]** 系统根据绑定域名生成专属探针代码。
* **[P29]** CLS 提供 JS 代码片段供复制。
* **[P30]** EBA 登录第三方建站系统，手动将代码嵌入网站 `<head>` 标签。
* **[P31]** 探针首次被访问触发后，系统校验请求来源域名是否与绑定域名一致，验证通过后标记代码嵌入成功。
* **[P32]** CLS 界面更新，向 EBA 显示「探针已生效，等待数据回传」。

### 🚫 异常处理机制

| 异常场景 | 系统行为 | 用户提示 |
| --- | --- | --- |
| 域名已绑定其他账号 | 拒绝绑定请求 | 「该域名已被其他账号绑定，如需协助请联系客服」 |
| 探针部署至未授权域名 | 拒绝数据上报 | 「检测到探针部署域名与绑定域名不一致，请检查部署位置」 |
| 账号已有绑定域名 | 阻止新域名绑定 | 「当前账号已绑定域名 xxx.com，如需更换请联系管理员」 |
| 域名格式无效 | 拒绝绑定 | 「请输入有效的网站域名，如 example.com」 |

# AI-LeadFinder 动态线索意图挖掘与全景解析 (P39-P46) 逻辑实现白皮书

本模块涵盖了系统**阶段三：访客追踪、条件识别与动态行为分析**的后半部分核心工程路径。引擎将通过高成本的三方 IP 解析服务 (IRS) 和联系人数据服务 (ECS)，把匿名的高分“访客表”记录，转化为实名的“动态线索”，并最终落库至**动态线索池 (Dynamic Lead Pool)**。

## 终局视角：P46 展示动态线索分析报告

**业务目标**：这是企业身份被揭开后的“真相时刻”。由于底层调用了相同的联系人接口，此页面的展示形态需要与 P27（静态线索报告）高度对齐，但在核心依据上，要用“动态行为”替代“静态海关/展会数据”。

* **UI 呈现核心元素**：
* **线索终极评分 (Final Dynamic Score)**：由基础行为分和核心决策人系数融合计算，如 `104/100 (S级极热线索)`。
* **企业身份与解析结果 (Resolution Result)**：
* *成功解析企业*：展示企业名、域名、行业。
* *解析为宽带(ISP)*：如 `Comcast Cable Communications`，展示 ⚠️ 提示：“该 IP 隶属于公共宽带，未能精准定位企业主体”。


* **原始动态行为证据 (Original Behavior Evidence)**：复刻 P38 里的"微观行为轨迹轴"，作为该线索高热度的铁证。
* **决策人矩阵列表 (Contact Matrix)**：展示获取到的关键联系人卡片（带邮箱/领英等标识）。
* **账单透明化**：展示本次 IP 解析与联系人挖掘的综合点数消耗。
* **行动召唤 (CTA)**：主按钮 `[进入智能触达 (ITA) 服务]`。

## P40：IP 解析与企业识别分流架构

**业务目标**：通过白嫖自有数据（UTM）或调用高成本外部接口，实现 IP 到 Domain 的映射。

* **🛠️ 核心执行链路**：
读取该访客日志中的 `traffic_context`。
**分支 A [P40]：存在有效 UTM 标识 (极低成本路径)**
* **触发条件**：如果访客 URL 带有 `utm_campaign=email_batch_01` 且带有我们 P58 触达时植入的 `user_id/lead_id`。
* **动作**：系统自动关联静态数据中已有的企业信息。跳过 IRS，直接去数据库匹配已知企业。`company_domain` = 库中已有域名。


**分支 B [P41]：无 UTM 标识，启动 IP 解析服务 (IRS)**
* **API 调用**：调用 IPinfo API。参数：`ip=67.43.156.99`。
* **处理逻辑**：
* `IF API返回 type="isp"` (如 Comcast 等公共网络)：标记 `resolve_status = "ISP"`, `domain = null`。
* `IF API返回 type="company"`：标记 `resolve_status = "Success"`, 提取 `company_domain` 和 `company_name`。

## P42 - P43：提交域名挖掘联系人数据 (ECS)

**业务目标**：与静态阶段复用底层基建，提交识别后的企业名称/域名，精准获取邮箱与社媒。

* **🛠️ 核心执行链路 (带阻断机制)**：
1. **拦截校验**：
`IF resolve_status == "ISP" 或者 domain == null`：**直接跳过 P43 的 ECS 调用**。不浪费 Apollo 积分，直接在 P44 输出空的联系人数组 `[]` 进入下一步。
2. **正常调用**：
`IF domain != null`：调用 Apollo `/v1/mixed_people/search` 接口，寻找 verified 邮箱。

## P44：AI 结合访客行为与联系人数据进行动态意向智能分析

**业务目标**：将"原始行为分"与"挖出的联系人质量"进行融合（乘数提权算法）。

* **📥 输入内容**：
1. P43 获取的 `contacts_list`。
2. P36 算出的原有的 `dynamic_behavior_score` (如：95分)。
3. P09 的 `ICP_LeadLevel` 基准。


* **🛠️ 模型选型**：`gpt-4o-mini`。
* **⚙️ 核心 Prompt (Markdown 格式)**：

```markdown
# Role
你是一位资深的 B2B 销售总监与动态线索质量评估专家。

# Input Data
- 访客原始行为分 (Base Behavior Score): {dynamic_behavior_score}
- 理想决策人画像 (ICP Lead Criteria): {icp_lead_criteria}
- 候选联系人列表: {contacts_list}
- IP 解析状态: {resolve_status}

# Task
1. **异常判定**：如果 {resolve_status} 为 "ISP" 或联系人列表为空，直接输出 final_dynamic_score = Base Behavior Score，并生成判词：“遗憾，该 IP 归属于公共宽带或未能找到决策人，但其行为数据真实有效，建议持续保持监测。”
2. **沙里淘金 (Ranking)**：若有联系人，评估其决策权，保留并排序核心联系人。
3. **执行乘数提权 (Apply Multiplier)**：
   - 根据你排在 No.1 的【核心决策人】职位提权：

   - 执行层 (Manager, Engineer): 系数 1.0
   - 普通级 (Info, Sales): 系数 0.8
   - 公式：最终动态线索分 = Base Behavior Score * 系数 (四舍五入取整)。
4. **生成 AI 综合判词**：结合前期的访客微观动作和本次挖出的人物，给销售明确的跟进建议。

# Output JSON Schema

  },
  "ai_reasoning": "该企业此前在独立站深度研究了您的 P1.9 防水屏参数。现已为您定位到其采购副总裁...这正是触达的最佳时机，跃升为 104 分 S 级线索。"
```

## P45：生成动态线索分析报告并更新至动态线索池 (含列表展示)

**业务目标**：完成资金清算，更新 P39 创建的占位记录状态，并在前端驱动【动态线索列表页】的实时刷新。

* **⚙️ 后台精算与异步状态更新**：
1. **价值精算 (Billing Logic)**：
* 固定扣除 3 Credits (用于 P41 的 IRS 成本)。
* 叠加计算联系人价值：`Sum (每人 2 分 + 社媒加分)`。若解析为 ISP 或未找到联系人，此处附加费为 0。


2. **状态更新 (State Update)**：
系统查找到 P39 插入的 `Resolving` 状态记录，将完整的报告数据（解析状态、联系人矩阵、最终得分、账单）**Update (更新)** 进去，并将记录状态修改为 `Completed (挖掘完成)`。


* **🖥️ 动态线索列表页界面说明 (Dynamic Lead Pool UI)**：
该页面是客户管理第一方动态流量的核心工作台，承载了历史线索的回溯与新线索的任务追踪。
* **进行中的任务 (Active Tasks)**：在列表最顶部，展示 P39 刚添加进来的线索。显示动画状态：“`🔍 正在深度挖掘 IP: 67.43.156.** 背后的企业与决策人...`”。
* **历史挖掘记录 (Historical Leads)**：下方按时间倒序展示已完成挖掘的动态线索记录。
* *成功高价值线索*：展示提取出的企业 Logo/名称、104分热度、挖出的关键联系人头像缩略图。
* *ISP 宽带线索*：保留 95 分的行为意向分，但企业名称处显示“ISP 匿名访客”，状态显示为“未查到对应企业”。（执行不抛弃原则，消除信息差）。
