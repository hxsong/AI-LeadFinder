## 二、文件结构层面问题

### 2.1 根目录文件扁平化，缺乏层级组织

**问题**：三个核心设计文件（修订日志、业务流、ER图）直接平铺在根目录，与项目配置文件混在一起。

**影响**：
- 项目根目录随迭代增加会越来越臃肿
- 新增概念设计类文件（如竞品分析、用户研究）无处可放，只能继续堆在根目录
- 后续查找文件效率下降

**建议**：建立 `docs/design/` 或 `docs/concepts/` 目录，统一收纳所有设计类文档（HTML + Markdown），根目录只保留启动脚本和项目索引文件。

---

### 2.2 PRD 文档目录结构过于简单

**问题**：`prd-md/` 下仅有 5 个 Markdown 文件和一个 api-specs 子目录，缺少上層索引文档。

**影响**：
- 无法快速了解 PRD 全貌（有哪些模块、覆盖哪些产品阶段）
- 新增 PRD 内容时没有规范的存放位置，只能追加到 prd-md/ 根目录
- API 规范和产品需求文档分离，但两者天然强关联（如"搜客"功能依赖出海派 API），却没有任何交叉引用机制

**建议**：在 `prd-md/` 下建立子目录结构，按产品阶段组织，子目录使用中文命名方便研发理解：

```
prd-md/
├── README.md                          # PRD 总索引
├── 00-概述与定义/                     # 产品定义、术语表
├── 01-账号与资源管理/                 # P01-P11 前置逻辑
├── 02-静态数据获客/                   # P12-P27
├── 03-动态流量获客/                   # P28-P46
├── 04-全景画像与触达/                 # P47-P70
├── 05-运营与合规/                     # P71-P80
├── 06-非功能需求/                     # 性能、安全、并发
├── 07-全局规范/                       # 响应码/Credits规则/AI异常处理/数据保留
└── api-specs/                        # 三方 API 集成规范
    ├── README.md                     # API 总览（新增）
    └── *.md
```

---

### 2.3 原型目录 `prototype/` 缺少索引和总览文件

**问题**：`prototype/` 下只有 `pages/` 和 `docs/` 两个子目录，没有任何索引文件说明：
- 原型覆盖了哪些产品阶段
- 每个页面与 PRD 中的哪个节点（P01-P80）对应
- 页面之间的跳转关系和入口出口

**影响**：
- 研发人员在开始开发前，必须阅读全部 22 个页面 + 22 个说明文档才能理解整体原型范围
- 无法快速定位"某个 PRD 节点对应的原型页面是哪个"
- 新增页面时没有规范约束，容易导致命名和目录混乱

**建议**：在 `prototype/` 下新增：
- `prototype/README.md` — 原型索引，列出所有页面对应的 PRD 节点、页面名称、跳转关系
- `prototype/CHANGELOG.md` — 页面变更日志（独立于根目录的修订日志.html）

---

### 2.4 修订日志 `LeadFinder修订日志.html` 与产品文档分离

**问题**：修订日志是 HTML 格式，放在根目录，而 PRD 是 Markdown 格式，放在 prd-md/。两者格式不统一，且修订日志仅记录了变更历史，缺少变更与 PRD 节点、页面之间的关联引用。

**影响**：
- 无法从 PRD 节点追溯到对应的修订记录
- 无法从修订记录快速跳转到被修改的 PRD 内容或页面
- HTML 格式的修订日志不便于在代码评审工具中 diff

**建议**：将修订日志迁移为 Markdown 格式，存放在 `docs/changelog/` 目录，或在 `prd-md/` 下增加修订记录章节。

## 三、内容结构层面问题

### 3.1 PRD 文档内容严重混杂，单文件体积极大

**问题**：`产品核心规则和实现思路.md` 一个文件超过 1300 行，包含了：
- P01-P03 账号初始化与多 BOT 逻辑
- P04-P11 业务初始化与分析流
- P12-P21 静态搜客与打分引擎
- P22-P27 静态线索深度挖掘
- P28-P38 动态流量捕捉
- P39-P46 动态线索意图挖掘

**影响**：
- 单文件过大导致编辑工具卡顿、Git diff 无法有效审查
- 各阶段内容没有独立的文件编号，引用困难（PRD 引用只能写"P28-P32"，无法直接链接到文件）
- 不同模块的负责人无法并行编辑

**建议**：每个 P 节点簇（阶段一、阶段二等）独立为单独文件，通过 `prd-md/README.md` 作为入口索引。

---

### 3.2 PRD 文档缺少结构化编号体系

**问题**：当前 PRD 文件没有采用层级编号系统（如 `PRD-01-概念设计`、`PRD-02-账号体系`、`PRD-03.1-搜客流程`）。

**影响**：
- 无法用编号快速引用具体需求（如"见 PRD-03.2.1"）
- 需求变更时无法精确标注影响范围
- 研发团队无法基于编号做需求管理（工时估算、依赖分析）

**建议**：结合子目录的中文命名，建立两级编号体系：`{中文子目录}/{序号}-{功能名}.md`，如 `02-静态数据获客/01-搜客需求输入.md`。在 `prd-md/README.md` 中建立完整的编号索引表。

---

### 3.3 原型页面与 PRD 节点没有建立映射

**问题**：22 个原型页面（`prototype/pages/`）对应 22 个功能说明文档（`prototype/docs/`），但没有任何文档说明这些页面对应 PRD 中的哪个节点。

**影响**：
- 研发人员必须同时打开 PRD Markdown + 原型 HTML + 说明文档 HTML 才能理解一个功能
- 需求评审时无法快速对齐"PRD 写的"和"原型展示的"是否一致
- 后续迭代时，无法判断修改某个 PRD 节点会影响哪些页面

**建议**：在 `prototype/README.md` 中建立映射表，作为研发快速定位的入口索引：

| 原型页面 | 功能说明 | 对应 PRD 节点 | 产品阶段 | 核心功能描述 |
|---------|---------|-------------|---------|------------|
| 搜客需求输入.html | 搜客需求输入-实现说明.html | P12 | 02-静态数据获客 | ... |
| 搜客策略预览.html | 搜客策略预览-实现说明.html | P13-P16 | 02-静态数据获客 | ... |

---

### 3.4 概念设计 HTML 与 PRD 文档脱节

**问题**：`LeadFinder业务流设计.html`（Mermaid 时序图）和 `产品核心业务流设计.md`（包含相同的 mermaid 时序图）是两份内容高度重复但格式不同的文件。

**影响**：
- 两份文件必须同步更新，人工维护成本高且容易产生不一致
- Mermaid 图在 Markdown 中的渲染依赖渲染器，在 HTML 中是直接嵌入的 SVG，两者的视觉效果可能不同

**建议**：统一使用 Markdown（Mermaid）作为唯一来源，HTML 文件通过渲染工具从 Markdown 生成。或者，只保留一份源文件（推荐 Markdown），删除重复的 HTML 文件。

---

### 3.5 API 规范文档缺少总览和使用指南

**问题**：`prd-md/api-specs/` 目录下有 7 个独立的 Markdown 文件，但：
- 没有 README.md 说明这些 API 的整体接入架构
- 没有 API 调用顺序或依赖关系的说明
- 每个 API 文档的格式和详细程度不一致（如有的写的是接口规格，有的写的是集成思路）

**影响**：
- 研发人员无法快速了解"这个系统需要接入哪些第三方服务，它们之间的关系是什么"
- 不同 API 文档的格式差异大，增加研发的理解成本
- 缺少 API 版本管理和变更记录

**建议**：在 `prd-md/api-specs/` 下增加 `README.md`，包含：
- 第三方服务总览表（服务名、用途、调用时机）
- API 调用流程图（哪个 PRD 节点调用哪个 API）
- 各 API 文档的格式规范（请求/响应/错误码模板）

---

### 3.6 功能说明文档（`prototype/docs/`）与 PRD 职责不清、内容重复

**问题**：功能说明文档（HTML）中的内容与 PRD 文档（Markdown）中的内容高度重叠。例如，`搜客需求输入-说明.html` 中描述的逻辑与 `产品核心规则和实现思路.md` 中 P12-P13 的描述几乎一致。两份文档的定位模糊，职责边界不清。

**影响**：
- 同一份信息维护在两个地方，迭代时容易遗漏同步
- 功能说明文档的读者不清楚自己究竟是产品经理、研发还是测试
- 文档职责不清晰：功能说明是"原型的注释"，还是"独立的需求规格"？

**解决方案：明确分工，各司其职（方向 B）**

将 `prd-md/` 和 `prototype/docs/` 定位为两种职责完全不同、互补且不重复的文档体系。

## 3. 不可见功能逻辑（后端处理）

### 3.1 前端请求
```javascript
// 调用接口
POST /api/v1/search/strategy/generate
{
  bot_id: string,
  user_query: string,
  use_default_icp: boolean
}
```

### 3.2 后端处理流程
服务端接收请求后的完整处理步骤：
1. 校验 bot_id 有效性
2. 若 user_query 为空，读取 P09 ICP 画像
3. 调用 LLM 生成搜客策略（gpt-4o）
4. 读取 API 单价表，计算预估消耗
5. 返回搜客策略 JSON

### 3.3 响应格式
```json
{
  "code": 200,
  "data": {
    "strategy_id": "STR-xxx",
    "estimated_credits": 148,
    "estimated_duration": "约 3 分钟",
    "execution_plan": [...]
  }
}
```

| 条件 | 系统行为 |
|-----|---------|

| 搜客进行中 | 禁止发起新的搜客任务（单 BOT 并发限制） |

## 5. API 调用清单

## 6. 异常处理

| 异常场景 | 前端表现 | 后端处理 |
|---------|---------|---------|
| 网络超时 | Toast 提示"网络异常，请重试" | 重试 1 次，失败返回错误码 |

| Credits 不足 | 按钮禁用 + 提示文案 | 不发起请求 |

## 7. 研发关注点

技术实现中的关键决策点、注意事项、可改进方向。
```

### 3.7 来自研发需求文档的补充内容（需纳入文档结构）

在审阅研发侧整理的《智能获客-需求文档V1.0.2.docx》后，发现当前文档体系中缺少以下内容，这些内容应作为独立章节或模板规范纳入文档体系：

#### 3.7.1 缺少「全局规范」章节

当前 `prd-md/` 和 `prototype/docs/` 均没有定义全局通用规范，导致同一规范在多个功能文档中重复描述或描述不一致。应在 `prd-md/` 下新增 `07-全局规范/` 目录，集中管理以下内容：

**① 全局响应码规范**

| 响应码 | 含义 | Toast 类型 | 说明 |
|--------|------|-----------|------|
| 200 | 操作成功 | success | 接口正常、业务成功 |
| 400 | 请求参数错误 | error | 前端传参有误 |
| 401 | 未登录/过期 | error | 清除 Token 后跳转登录页 |
| 403 | 无权限 | warn | |
| 404 | 接口不存在 | error | |
| 500 | 服务器异常 | error | 后端代码/服务报错 |
| 1001 | 业务提示 | info | 普通提示说明类信息 |
| 1002 | 业务警告 | warn | 风险提示、非强制错误 |
| 1003 | 加载中状态 | loading | 异步处理、上传/下载中 |

**② 余额（Credits）规则**

| 场景 | 规则 |
|------|------|
| 余额可为负 | 允许余额为负数 |
| 操作限制 | 余额 > 0 时允许操作；余额 ≤ 0 时禁止所有消耗操作 |
| 预估消耗检查 | 有预估消耗的场景需余额 ≥ 预估消耗 |
| 执行中不扣费 | 任务执行过程中不扣除 Credits，完成后按实际消耗结算 |
| 完成后结算 | 任务完成后统一结算，允许余额为负 |

**③ 全局异常处理策略**

AI 业务分析异常处理（各模块通用）：

| 异常类型 | 触发条件 | 处理策略 |
|---------|---------|---------|
| 输出为空 | AI 返回空结果 | 自动重试 1 次，失败后使用规则评分/默认值 |
| 格式校验失败 | AI 返回 JSON 解析失败 | 自动重试 1 次，失败后使用默认值 |
| 一致性校验失败 | 评分与原始数据矛盾 | 切换备用模型重新评分 |
| 数值范围超限 | 评分超出 0-100 范围 | 使用默认值（边界值） |
| 字段缺失 | 必要字段为空 | 提示数据不完整，部分内容缺失 |

**④ 数据保留策略**

| 数据类型 | 保留时长 | 说明 |
|---------|---------|------|
| 搜客记录 | 90 天 | 超期自动归档 |
| 访客表数据 | 90 天 | 超期自动归档，已挖掘访客数据永久保留 |
| 已挖掘动态线索 | 永久 | 企业信息永久保留 |
| 联系人数据 | 永久 | 重新挖掘后覆盖 |
| 业务分析报告 | 永久 | 作为 ICP 画像基础数据 |

## 四、样式和技术层面问题

### 4.1 原型页面样式不统一

**问题**：不同原型页面使用了不同的 Tailwind 配色和字体配置：
- `prototype/pages/访客雷达.html`：primary `#0F172A`，字体 Plus Jakarta Sans
- `prototype/pages/多BOT工作台.html`：primary `#3B82F6`，字体 Fira Sans
- `prototype/docs/搜客需求输入-说明.html`：primary `#3B82F6`，字体 Fira Sans

**影响**：
- 用户在浏览不同页面时视觉体验不一致
- 后续维护时没有统一的组件规范可参考
- 不同页面的组件无法复用（虽然都是 HTML，但样式定义散落在各处）

**建议**：
1. 建立统一的 `prototype/styles/` 目录，存放全局样式文件（Tailwind 配置、设计 token）
2. 所有页面引用统一的样式文件，而非在每个 HTML 中重复定义
3. 建立组件库（Button、Card、Modal 等的 HTML 模板），减少重复代码

### 4.2 左侧导航 `entry-nav.js` 硬编码了页面列表

**问题**：`prototype/entry-nav.js` 中的页面列表是硬编码的：

const entryPages = [
    { name: "搜索客户", file: "搜客需求输入.html", icon: "search" },
    { name: "访客雷达", file: "访客雷达.html", icon: "radar" },

**影响**：

- entry-nav.js 只包含了 5 个核心页面，22 个页面中大部分没有在左侧导航中暴露入口

**建议**：将 entry-nav.js 改造为自动扫描 `prototype/pages/` 目录动态生成导航，或者将导航配置提取到独立的 `prototype/nav-config.json` 文件中管理。
