## 数据契约

### 页面展示数据

| 前端展示字段 | 后端字段名 | 类型 | 来源/存储 | 说明 |
|-------------|-----------|------|----------|------|
| 任务 ID | `task_id` | `string` | 路由参数传入 | 系统生成 UUID |
| BOT ID | `bot_id` | `string` | 路由参数传入 | 当前 BOT 唯一标识 |
| 输入数据来源 | `source` | `string` | 路由参数传入 | `upload`（文档）/ `site`（独立站） |

| 阶段状态 | `stages[].status` | `string` | 接口返回 | `pending` / `in_progress` / `completed` / `failed` |
| 阶段进度 | `stages[].progress` | `number` | 接口返回 | 0-100 整数 |
| 阶段输出摘要 | `stages[].output_summary` | `string` | 接口返回 | AI 处理结果摘要，可能为 null |
| 整体状态 | `overall_status` | `string` | 接口返回 | `processing` / `completed` / `failed` |
| 预计完成时间 | `estimated_completion` | `string` | 接口返回 | ISO 8601 时间戳 |

## 页面流转

### 入口定义

| 来源 | 触发条件 | 携带参数 |
|------|---------|---------|
| 上传业务材料.html | 所有文件上传成功后自动跳转 | `bot_id`, `task_id`, `source=upload` |
| 独立站授权.html | 授权成功后自动跳转 | `bot_id`, `task_id`, `source=site` |

### 出口定义

| 目标 | 触发条件 | 携带数据 |
|------|---------|---------|
| 业务分析报告.html | 所有阶段处理完成后自动跳转 | `bot_id`, `task_id` |

### 页面状态

| 状态 | 说明 | 进入条件 | 退出条件 |
|------|------|---------|---------|
| 初始化中 | 页面加载，获取任务状态 | 页面加载完成 | 首次轮询请求发出 |
| 等待中 | 任务未开始或处于早期阶段 | `overall_status` 为 `pending` | 任一阶段进入 `in_progress` |
| P07 进行中 | 文本提取与结构化执行中 | `stages[0].status` 为 `in_progress` | P07 状态变为 `completed` 或 `failed` |
| P07 完成 | 文本提取与结构化完成 | `stages[0].status` 为 `completed` | P08 进入 `in_progress` |
| P08 进行中 | 深度分析与 ICP 建模执行中 | `stages[1].status` 为 `in_progress` | P08 状态变为 `completed` 或 `failed` |
| P08 完成 | 深度分析与 ICP 建模完成 | `stages[1].status` 为 `completed` | 1.5s 后自动跳转 |
| P09 进行中 | 报告生成中（原型展示） | `stages[2].status` 为 `in_progress` | P09 状态变为 `completed` |
| 处理完成 | 所有阶段执行完毕 | `overall_status` 为 `completed` | 页面跳转 |
| 失败 | 任一阶段执行失败 | 任一 `stages[].status` 为 `failed` | 用户点击重试或返回 |

## 处理逻辑

### 场景一：文档上传模式 - 文本提取与结构化（P07）

**触发条件**：用户从上传业务材料.html 进入，即 `source=upload`。

**执行逻辑**：
1. 系统接收上传文件：
   - PDF 格式：调用 LlamaParse 或 Unstructured.io 解析
   - WORD 格式（DOC/DOCX）：使用 python-docx 解析
   - MD/TXT 格式：直接读取文本内容
2. 系统输出带排版噪音的 Markdown 长文本
3. 系统调用 ETL 数据清洗 Prompt：
   - 模型：`kimi-k2.5` / `qwen-max`（优先国内模型）/ `gpt-4o-mini`（备选）
   - 强制开启 JSON Mode
4. AI 严格按 JSON Schema 输出干净的实体字典：
   - 遵循零捏造原则
   - 空值处理：无法提取的字段返回空字符串或空数组
   - 去营销化：去除夸张宣传用语，保留客观事实
5. 系统校验必填字段完整性：
   - 必填字段：企业名称、企业定位、业务类型、主营品类、目标客户类型、目标客户市场区域
   - 若必填字段缺失，中断流程，提示客户补充信息，不进入 P08

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 文件解析失败 | 标记 P07 失败，返回错误码 500，前端展示重试按钮 |
| AI 输出为空 | 自动重试 1 次，仍为空则标记失败 |
| AI 格式错误 | 自动重试 1 次，仍错误则标记失败 |
| 必填字段缺失 | 中断流程，不进入 P08，前端提示补充字段列表 |
| 可选字段缺失 | 使用默认值或 AI 推断，不阻断流程 |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| 等待中 | P07 进行中 | 任务开始执行 |
| P07 进行中 | P07 完成 | 文本提取成功且必填字段完整 |
| P07 进行中 | 失败 | 文本提取失败或必填字段缺失 |

**输出**：干净的实体字典 JSON，存储至 MySQL。

### 场景二：独立站授权模式 - 数据解析（P07）

**触发条件**：用户从独立站授权.html 进入，即 `source=site`。

**执行逻辑**：
1. 系统从已同步的站点数据中提取原始内容：
   - 网站 title、description、产品页面文本等
2. 系统将提取的原始内容组装为 Markdown 格式长文本
3. 系统调用 ETL 数据清洗 Prompt（与文档上传模式一致）：
   - 模型：`kimi-k2.5` / `qwen-max`（优先国内模型）/ `gpt-4o-mini`（备选）
   - 强制开启 JSON Mode
4. AI 严格按 JSON Schema 输出干净的实体字典（零捏造原则，空值处理，去营销化）
5. 系统校验必填字段完整性：
   - 必填字段：企业名称、企业定位、业务类型、主营品类、目标客户类型、目标客户市场区域
   - 若必填字段缺失，中断流程，提示客户补充信息，不进入 P08

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 站点数据未同步 | 等待同步完成后执行，或标记失败 |
| 站点数据提取为空 | 标记 P07 失败，返回错误码 500 |
| AI 输出为空 | 自动重试 1 次，仍为空则标记失败 |
| AI 格式错误 | 自动重试 1 次，仍错误则标记失败 |
| 必填字段缺失 | 中断流程，不进入 P08，前端提示补充字段列表 |
| 可选字段缺失 | 使用默认值或 AI 推断，不阻断流程 |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| 等待中 | P07 进行中 | 站点数据已同步，开始 ETL 处理 |
| P07 进行中 | P07 完成 | ETL 处理成功且必填字段完整 |
| P07 进行中 | 失败 | 站点数据缺失或 ETL 处理失败 |

**输出**：干净的实体字典 JSON，存储至 MySQL。

### 场景三：深度分析与 ICP 建模（P08）

**触发条件**：P07 阶段完成，且必填字段完整。

**执行逻辑**：
1. 系统将 P07 输出的实体字典输入 AI 分析 Prompt：
   - 模型：`kimi-k2.5` / `qwen-max`（优先国内模型）/ `claude-3.5-sonnet`（备选）
2. AI 综合分析输出：
   - 企业定位确认
   - 业务类型识别（外贸型/生产型/服务型/渠道型）
   - 主营品类细化
   - 核心优势提炼
   - 供应链分析（上游依赖、自身生态位、下游潜客群）
   - 目标客户画像（类型、市场区域、主营品类）
3. 系统根据业务类型生成动态四维评分权重：
   - 外贸型：需求匹配度 32% / 购买能力 20% / 采购意向 35% / 活跃意向 13%
   - 生产型：需求匹配度 40% / 购买能力 25% / 采购意向 20% / 活跃意向 15%
   - 服务型：需求匹配度 35% / 购买能力 25% / 采购意向 15% / 活跃意向 25%
   - 渠道型：需求匹配度 35% / 购买能力 20% / 采购意向 30% / 活跃意向 15%
4. 系统将客户信息和评分模型存储至 MySQL，供后续搜客评分计算使用

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| AI 输出为空 | 自动重试 1 次，仍为空则标记失败 |
| AI 格式错误 | 自动重试 1 次，仍错误则标记失败 |
| AI 内容幻觉 | 标记异常，前端提示建议重新分析 |
| AI 字段缺失 | 使用默认值填充，前端提示数据不完整 |
| AI 评分异常 | 权重超出合理范围时，使用业务类型默认值替代 |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| P07 完成 | P08 进行中 | P07 输出校验通过 |
| P08 进行中 | P08 完成 | ICP 建模成功，数据已存储 |
| P08 进行中 | 失败 | ICP 建模失败或存储异常 |

**输出**：完整 ICP 画像及评分模型，存储至 MySQL。

### 场景四：处理进度实时展示

**触发条件**：页面加载后，异步任务执行中。

**执行逻辑**：
1. 页面加载时解析 URL 参数，获取 `bot_id`、`task_id`、`source`
2. 根据 `source` 值切换前端文案：
   - `upload`：展示"文本内容提取与结构化"相关文案
   - `site`：展示"数据解析与结构化"相关文案
3. 页面发起轮询请求获取任务状态，默认间隔 2 秒
4. 系统根据返回的 `stages` 数据渲染进度卡片：
   - 等待中状态：灰色圆圈 + "等待中"标签
   - 执行中状态：蓝色旋转图标 + "进行中"标签 + 进度条动画
   - 已完成状态：绿色勾选图标 + "已完成"标签
   - 失败状态：红色叉号图标 + "失败"标签 + 重试按钮
5. 进度条动画：根据 `progress` 值平滑递增，带波浪流动效果
6. 中心动画：三层同心圆，最内层旋转，外层脉冲扩散
7. 页面组件依次淡入上滑（fadeInUp）
8. 所有步骤完成后，延迟 1.5 秒自动跳转至业务分析报告页面

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 轮询请求超时 | 继续下一次轮询，前端保持当前状态 |
| 网络断开 | 停止轮询，前端展示网络异常提示 |
| 用户关闭页面 | 异步任务继续在后台执行，不影响处理 |
| 跳转失败 | 前端展示手动跳转按钮 |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| 初始化中 | 等待中 | 任务状态返回 pending |
| 等待中 | P07 进行中 | stages[0].status 变为 in_progress |
| P07 进行中 | P07 完成 | stages[0].status 变为 completed |
| P07 完成 | P08 进行中 | stages[1].status 变为 in_progress |
| P08 进行中 | P08 完成 | stages[1].status 变为 completed |
| P08 完成 | 处理完成 | overall_status 变为 completed |
| 处理完成 | 已跳转 | 1.5s 延迟后执行页面跳转 |

**输出**：实时进度 UI，处理完成自动跳转。

## 异常兜底

| 异常场景 | 检测方式 | 后端处理 | 前端表现 |
|---------|---------|---------|---------|
| 任务不存在 | HTTP 404 | 返回 404 错误码 | 跳转至业务初始化页面，Toast：「任务不存在或已过期，请重新开始」 |
| P07 文本提取失败 | 解析服务返回错误 | 标记 P07 失败，记录错误日志 | 失败提示 + 重试按钮，Toast：「文本提取失败，请检查上传的文件」 |
| P08 ICP 建模失败 | AI 服务返回错误或超时 | 标记 P08 失败，记录错误日志 | 失败提示 + 重试按钮，Toast：「ICP 建模失败，请重试」 |
| 任务超时 | 10 分钟未完成的定时任务检测 | 自动标记任务为失败 | 超时提示，Toast：「任务处理超时，请稍后重试」 |
| AI 输出为空 | 校验返回数据为 null | 自动重试 1 次，仍为空则标记失败 | Toast：「AI 分析结果为空，正在重试...」 |
| AI 格式错误 | JSON 解析失败 | 自动重试 1 次，仍错误则标记失败 | Toast：「AI 返回数据格式异常，正在自动重试...」 |
| AI 内容幻觉 | 与输入材料矛盾（语义相似度检测） | 标记异常，记录告警 | Toast：「AI 分析结果存在异常，建议重新分析」 |
| AI 字段缺失 | 必要字段为空 | 使用默认值填充 | Toast：「AI 分析不完整，部分数据缺失」 |
| AI 评分异常 | 权重超出 0-1 范围或不满足和为 1 | 使用业务类型默认权重替代 | Toast：「AI 评分异常，已使用默认值」 |
| 网络超时 | 请求超时（>10s 无响应） | 返回 504 或前端超时处理 | 错误提示，Toast：「网络连接异常，请检查网络后重试」 |
| 服务不可用 | HTTP 500 / 503 | 返回对应错误码，触发告警 | 全局错误处理，Toast：「服务暂时不可用，请稍后再试」 |
| 跳转失败 | 页面跳转异常捕获 | - | 展示手动跳转按钮，Toast：「页面跳转失败，请手动继续」 |
| P07 必填字段缺失 | AI 输出校验 | 中断流程，不进入 P08 | 中断提示，Toast：「部分必填字段缺失，请返回补充：{fields}」 |
| 并发冲突 | 同一租户存在进行中的分析任务 | 拒绝新任务，返回 409 | Toast：「已有分析任务进行中，请等待完成」 |

## AI 提示词

### P07 文本提取与结构化 Prompt

**模型**：`kimi-k2.5` / `qwen-max`（优先国内模型）/ `gpt-4o-mini`（备选）

**强制设置**：`response_format: { "type": "json_object" }`

**系统提示词**：

```
你是一位专业的企业信息提取专家。你的任务是从用户上传的业务文档中提取结构化的企业信息。

## 提取原则
1. 零捏造原则：只提取文档中明确出现的信息，绝不编造
2. 去营销化：去除夸张宣传用语，保留客观事实
3. 空值处理：无法提取的字段返回空字符串或空数组

## 输出格式
必须严格按照以下 JSON Schema 输出：
{
  "企业名称": "string",
  "企业定位": "string",
  "业务类型": "string (外贸型/生产型/服务型/渠道型)",
  "主营品类": ["string"],
  "核心优势": ["string"],
  "企业规模": {
    "员工规模": "string",
    "营收规模": "string"
  },
  "供应链分析": {
    "上游依赖": ["string"],
    "自身生态位": "string",
    "下游潜客群": ["string"]
  },
  "目标客户类型": ["string"],
  "目标客户市场区域": ["string"],
  "目标客户主营品类": ["string"]
}

## 必填字段
以下字段必须提取到有效值，否则返回空值：
- 企业名称
- 企业定位
- 业务类型
- 主营品类
- 目标客户类型
- 目标客户市场区域
```

**用户提示词模板**：

```
请从以下文档内容中提取企业信息：

[文档内容]
{{document_text}}
[/文档内容]

请严格按照系统提示词中的 JSON Schema 返回结果。
```

### P08 深度分析与 ICP 建模 Prompt

**模型**：`kimi-k2.5` / `qwen-max`（优先国内模型）/ `claude-3.5-sonnet`（备选）

**系统提示词**：

```
你是一位资深的商业分析师，擅长企业商业模式分析和目标客户画像构建。

## 任务目标
基于已提取的企业结构化信息，进行深度商业分析，输出完整的 ICP（理想客户画像）。

## 分析维度
1. 企业定位确认与优化
2. 业务类型识别与验证（外贸型/生产型/服务型/渠道型）
3. 主营品类细化与分类
4. 核心优势提炼（最多5条）
5. 供应链上下游分析
6. 目标客户画像构建

## 业务类型判定规则
- 外贸型：以进出口贸易为主，有明确的海外客户群体
- 生产型：以产品制造为主，拥有生产线或工厂
- 服务型：以服务交付为主，如咨询、设计、物流等
- 渠道型：以分销代理为主，连接上下游资源

## 四维评分权重配置
根据业务类型自动配置权重：
- 外贸型：需求匹配度 32% / 购买能力 20% / 采购意向 35% / 活跃意向 13%
- 生产型：需求匹配度 40% / 购买能力 25% / 采购意向 20% / 活跃意向 15%
- 服务型：需求匹配度 35% / 购买能力 25% / 采购意向 15% / 活跃意向 25%
- 渠道型：需求匹配度 35% / 购买能力 20% / 采购意向 30% / 活跃意向 15%

## 输出格式
{
  "customer_info": {
    "company_name": "string",
    "company_positioning": "string",
    "business_type": "string",
    "main_categories": ["string"],
    "core_advantages": ["string"],
    "company_scale": {
      "员工规模": "string",
      "营收规模": "string"
    },
    "supply_chain": {
      "上游依赖": ["string"],
      "自身生态位": "string",
      "下游潜客群": ["string"]
    },
    "target_customer_types": ["string"],
    "target_markets": ["string"],
    "target_categories": ["string"]
  },
  "scoring_model": {
    "business_type": "string",
    "weights": {
      "需求匹配度": number,
      "购买能力": number,
      "采购意向": number,
      "活跃意向": number
    }
  }
}
```

**用户提示词模板**：

```
请基于以下企业信息进行深度分析和 ICP 建模：

[企业基础信息]
{{p07_output_json}}
[/企业基础信息]

请输出完整的客户画像和评分模型配置。
```
