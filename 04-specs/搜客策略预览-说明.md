## 数据契约

### 页面展示数据

| 前端展示字段 | 后端字段名 | 类型 | 来源/存储 | 说明 |
| ------------ | --------- | ---- | -------- | ---- |
| BOT ID | `bot_id` | `string` | 路由参数传入 | 当前 BOT 唯一标识，从 URL 参数 `?bot_id=xxx` 获取 |
| 任务 ID | `task_id` | `string` | 路由参数传入 | 搜客任务唯一标识，从 URL 参数 `?task_id=xxx` 获取 |
| 策略 ID | `strategy_id` | `string` | 系统生成 | AI 生成的策略唯一标识，格式：`STRAT-{YYYYMMDD}-{6位序号}` |
| 策略摘要 | `summary` | `string` | AI 生成 | 策略摘要说明（50字以内），由 AI 根据用户需求和 ICP 画像生成 |
| 数据源名称 | `data_sources[].name` | `string` | AI 生成/系统追加 | 数据源名称：出海派-海关/出海派-展会/Google Maps/Jina Reader。主数据源由 AI 根据意图匹配唯一选择；**Jina Reader 由后端系统强制追加，对每家企业循环进行网页深度验证** |
| 数据源优先级 | `data_sources[].priority` | `number` | AI 生成/系统追加 | 执行优先级排序：主数据源为 1，Jina Reader 系统强制深度验证为 2 |
| 预估结果数 | `data_sources[].estimated_results` | `number` | AI 估算/系统计算 | 预计返回企业数量。主数据源由 AI 估算；Jina Reader 由系统计算 = 主数据源预估结果数 |
| 预估 MoliAI点数 | `data_sources[].estimated_moli_ai_points` | `number` | 后端计算 | 预计消耗 MoliAI点数，计算公式：单价(MoliAI点数/次) × 调用次数，详见「技术处理逻辑-费用计算逻辑」 |
| 瀑布流步骤 | `waterfall[].step` | `number` | AI 编排/系统追加 | 步骤序号，从 1 开始递增。主数据源步骤由 AI 生成并逐条罗列；**最后一步 Jina Reader 由后端系统强制追加，对每家企业循环验证** |
| 步骤数据源 | `waterfall[].source` | `string` | AI 生成/系统追加 | 该步骤调用的数据源名称。主数据源步骤由 AI 生成；Jina Reader 步骤由系统硬编码追加 |
| 执行筛选条件 | `waterfall[].filters` | `object` | AI 生成 | 各步骤筛选条件，包含接口调用参数 |
| 基础费用 | `cost_estimate.base` | `number` | 后端计算 | API 基础调用费用，单一主数据源 + Jina Reader 预估 MoliAI点数 之和 |
| 动态溢价 | `cost_estimate.premium` | `number` | 后端计算 | base × 30%，用于覆盖实际调用波动 |
| 总预估费用 | `cost_estimate.total` | `number` | 后端计算 | base + premium，用户看到的预估总消耗 |
| 执行后余额 | `moli_ai_balance_after` | `number` | 后端计算 | 执行后预估余额 = 当前余额 - total |
| MoliAI点数余额 | `moli_ai_balance` | `number` | 接口返回 | 当前租户 MoliAI点数余额，从 `moli_ai_balances` 表查询 |
| 目标企业数量 | `target_count` | `number` | 数据库查询 | 从 `search_tasks.target_count` 读取用户设置的目标数量 |
| 预计候选企业 | `estimated_candidates` | `number` | 后端计算 | 单一主数据源与 Jina Reader 预估结果数之和 |
| 最终输出企业 | `final_output_count` | `number` | 后端计算 | 等于 `target_count`，展示给用户的目标数量 |

## 页面流转

### 入口定义

| 来源页面 | 触发条件 | 携带参数 |
|----------|----------|----------|
| 搜客需求输入.html | 用户点击"确认搜客"按钮提交成功 | `bot_id`, `task_id` |

**入口前置条件**：

- 用户已登录（携带有效 `tenant_token`）
- URL 中包含有效的 `bot_id` 和 `task_id` 参数
- 搜客任务已创建且状态为 `pending`

### 出口定义

| 目标页面 | 触发条件 | 携带数据 |
|----------|----------|----------|
| 搜客执行中.html | 用户确认策略后点击"确认开始搜客" | `bot_id`, `task_id` |
| 搜客需求输入.html | 用户点击"修改策略"按钮 | `bot_id` |

### 页面状态及流转条件

| 状态 | 说明 | 进入条件 | 退出条件 |
|------|------|----------|----------|
| 加载中 | AI 策略生成中 | 页面加载，调用策略生成接口 | 策略生成完成或失败 |
| 正常 | 策略生成成功，展示策略内容 | 策略接口返回成功 | 用户点击执行或返回 |
| 策略异常 | AI 策略生成失败或超时 | 策略接口返回失败或超时 | 用户点击重试或返回 |
| 执行中 | 用户已点击确认执行 | 用户点击"确认开始搜客" | 执行接口返回成功或失败 |
| 空数据 | 策略生成成功但数据为空 | 策略接口返回成功但 data 为空 | 用户点击返回修改 |

**状态流转图**：

```
加载中 --(生成成功)--> 正常 --(点击执行)--> 执行中 --(成功)--> 搜客执行中.html
加载中 --(生成失败)--> 策略异常 --(点击重试)--> 加载中
加载中 --(生成成功但空数据)--> 空数据 --(点击返回)--> 搜客需求输入.html
正常 --(点击返回)--> 搜客需求输入.html
```

## 业务处理逻辑

### 场景一：AI 策略生成

**触发条件**：页面加载时，后端基于搜客需求输入生成 AI 策略。

**执行逻辑**：

1. 系统接收请求：页面加载，携带 `bot_id` 和 `task_id`
2. 系统执行处理：
   - 前端调用 `POST /api/v1/search/strategy/generate`
   - 后端调用 AI 策略生成接口
   - AI 分析用户需求 + ICP 画像，输出结构化策略 JSON
   - 前端解析策略，展示三大模块：策略解读、瀑布流执行计划、费用预估明细、目标对齐
   - 页面组件依次淡入上滑（fadeInUp），带延迟效果（100ms-450ms）
   - 顶部显示"AI 策略规划完成"绿色徽章
3. 系统返回响应：返回策略详情数据

**边界条件**：

| 条件 | 系统行为 |
|------|----------|
| AI 策略生成超时（60s） | 降级返回预置策略模板，Toast「AI 分析超时，已使用默认策略」 |
| AI 输出为空 | 提示"AI 策略生成结果为空，请重试"，显示重试按钮 |
| JSON 格式错误 | 自动重试 1 次，Toast「AI 返回数据格式异常，正在自动重试...」 |
| 策略与需求矛盾 | 显示警告"AI 策略与需求不符，建议重新生成"，显示重新生成按钮 |
| 必要字段缺失 | 自动重试 1 次，若仍缺失则返回错误 |
| 数据源名称非法 | 若主数据源名称非法，返回错误要求重新生成；若 Jina Reader 标识异常，仍继续执行但标记深度验证步骤异常 |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| 加载中 | 正常 | 策略生成成功 |
| 加载中 | 策略异常 | 策略生成失败或超时 |
| 加载中 | 空数据 | 策略生成成功但 data 为空 |
| 策略异常 | 加载中 | 用户点击重试 |

**输出**：策略解读 + 瀑布流 + 费用预估 + 目标对齐四大模块。

### 场景二：策略解读展示

**触发条件**：AI 策略生成完成，进入正常状态。

**执行逻辑**：

1. 系统接收请求：策略数据加载完成
2. 系统执行处理：
   - 展示 AI 生成的策略摘要文字（`summary` 字段）
   - 展示 AI 生成的单一主数据源类型（海关数据 / 展会数据 / 地图数据，三选一）
   - 展示每个主数据源步骤的筛选条件说明
   - 若主数据源离线，标记该步骤异常状态
3. 系统返回响应：策略解读模块渲染完成

**边界条件**：

| 条件 | 系统行为 |
|------|----------|
| 数据源离线 | 对应步骤标记异常状态，不阻断其他步骤 |
| 策略摘要为空 | 显示默认摘要"基于 ICP 画像的默认搜客策略" |
| 策略摘要超过 50 字 | 截断显示前 50 字，悬浮显示完整内容 |

**状态变更**：无

**输出**：策略解读模块。

### 场景三：瀑布流执行计划展示

**触发条件**：策略解读展示完成。

**执行逻辑**：

1. 系统接收请求：策略数据已加载
2. 系统执行处理：
   - 展示执行计划瀑布流图（数据获取步骤串联）
   - 每个步骤显示：步骤序号、数据类型（如海关数据/展会数据）、预计返回数量、预估 MoliAI点数消耗
   - 展示执行顺序说明（串行执行，先主数据源后深度验证）
   - **根据用户意图只匹配单一主数据源**，不再混合调用多个接口
   - **网页深度验证（Jina Reader）作为系统强制执行的最终步骤**：由后端系统自动追加到 waterfall 末尾，对主数据源获取的每一家企业循环进行网页抓取深度验证
   - **瀑布流步骤逐条罗列**：主数据源根据实际调用次数逐条展开（如 Google Maps 单次限制 20 条，目标 100 家则拆分为 5 个步骤，每个步骤携带具体传参），最后一步固定为系统强制的 Jina Reader 深度验证
   - Jina Reader 步骤前端特殊标注为"系统强制验证"，表明该步骤不由 AI 生成而是由系统硬编码执行
   - 每个步骤展示详细的接口调用参数（filters 内容）
3. 系统返回响应：瀑布流执行计划模块渲染完成

**边界条件**：

| 条件 | 系统行为 |
|------|----------|
| 主数据源步骤数过多（>10） | 前端折叠展示，点击展开查看全部参数 |
| 某步骤数据源离线 | 该步骤标记为异常，Jina Reader 步骤仍正常展示 |
| filters 参数为空 | 显示"使用默认筛选条件" |

**状态变更**：无

**输出**：瀑布流执行计划模块。

### 场景四：费用预估明细展示

**触发条件**：瀑布流展示完成。

**执行逻辑**：

1. 系统接收请求：策略数据已加载
2. 系统执行处理：
   - 展示预估费用明细表：
     - 各 API 基础调用费用（按数据源分组）
     - 动态溢价（30% buffer）
     - 总预估费用（MoliAI点数）
   - 展示 MoliAI点数余额和执行后预估余额
   - 余额规则：可为负，≤0 时禁止消耗操作；执行中不扣费，完成后按实际消耗结算
   - 显示费用说明提示："实际扣费将按真实检索成功量结算，多退少补"
3. 系统返回响应：费用预估模块渲染完成

**边界条件**：

| 条件 | 系统行为 |
|------|----------|
| MoliAI点数余额 < 预估消耗 | "确认执行"按钮禁用，提示充值 |
| 余额 ≤ 0 | 禁止消耗操作，按钮禁用 |
| 预估费用为 0 | 正常显示，提示"本次搜客免费" |

**状态变更**：无

**输出**：费用预估模块。

### 场景五：目标对齐展示

**触发条件**：费用预估展示完成。

**执行逻辑**：

1. 系统接收请求：策略数据已加载
2. 系统执行处理：
   - 展示目标对齐信息：
     - 目标企业数量（用户设置的 target_count）
     - 预计候选企业（单一主数据源预估结果数，Jina Reader 对每一家进行深度验证）
     - 最终输出企业（等于目标企业数量）
   - 显示说明文字："预计候选企业 = 主数据源预估获取数量（经网页深度验证）"
3. 系统返回响应：目标对齐模块渲染完成

**边界条件**：

| 条件 | 系统行为 |
|------|----------|
| 预计候选企业 < 目标企业数量 | 显示警告"预计获取数量可能不足，建议调整需求" |
| 预计候选企业 = 0 | 显示警告"暂无可匹配企业，建议调整筛选条件" |

**状态变更**：无

**输出**：目标对齐模块。

### 场景六：确认执行

**触发条件**：用户确认策略无误，点击"确认开始搜客"按钮。

**执行逻辑**：

1. 系统接收请求：用户点击"确认开始搜客"按钮
2. 系统执行处理：
   - 前端显示"冻结 MoliAI点数..."加载状态
   - 按钮禁用，防止重复提交
   - 调用 `POST /api/v1/search/execute`，传入 `bot_id`、`task_id`、`strategy_id`
   - 接口返回成功后跳转至搜客执行中页面
3. 系统返回响应：返回执行结果和跳转地址

**边界条件**：

| 条件 | 系统行为 |
|------|----------|
| MoliAI点数余额不足 | 按钮禁用，Toast「MoliAI点数余额不足，无法执行」 |
| 策略已过期（>30分钟） | Toast「策略已过期，请重新生成」 |
| 重复点击 | 按钮 loading 状态防止重复提交 |
| 网络超时 | Toast「网络连接异常，请检查网络后重试」，支持重试 |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| 正常 | 执行中 | 用户点击"确认开始搜客" |
| 执行中 | 正常 | 执行接口返回失败 |
| 执行中 | 跳转中 | 执行接口返回成功 |

**输出**：跳转至搜客执行中页面。

## 技术处理逻辑

### 数据加工规则

| 前端字段 | 加工类型 | 技术处理逻辑 | 存储位置 |
|----------|----------|----------------|----------|
| 策略 ID | 系统生成 | 服务端生成唯一标识，格式：`STRAT-{YYYYMMDD}-{6位序号}` | MySQL `search_strategies` 表 |
| 策略摘要 | AI 生成 | 调用 AI 模型（Kimi k2.5 / 通义千问 qwen-max）生成，基于 ICP 画像和用户需求 | MySQL `search_strategies.summary` |
| 数据源列表 | AI 生成 + 系统追加 | AI 只生成单一主数据源；**Jina Reader 由后端系统强制追加**，对每家企业循环深度验证。计算优先级和预估结果数 | MySQL `search_strategies.data_sources` |
| 瀑布流步骤 | AI 编排 + 系统追加 | AI 生成主数据源执行顺序并逐条罗列（如 Google Maps 目标 100 家拆为 5 个步骤）；**最后一步 Jina Reader 由系统硬编码追加** | MySQL `search_strategies.waterfall` |
| 预估 MoliAI点数 | 后端计算 | 根据各数据源定价规则、接口限制和目标数量计算：单价 × 调用次数，详见「费用计算逻辑」 | MySQL `search_strategies.data_sources` |
| 基础费用 | 后端计算 | 单一主数据源 + Jina Reader 预估 MoliAI点数 之和 | 内存计算，不存储 |
| 动态溢价 | 后端计算 | base × 0.3（30% buffer） | 内存计算，不存储 |
| 总预估费用 | 后端计算 | base + premium | 内存计算，不存储 |
| MoliAI点数余额 | 数据查询 | 从 `moli_ai_balances` 表查询 `balance` 字段 | MySQL `moli_ai_balances` 表 |
| 执行后余额 | 后端计算 | moli_ai_balance - cost_estimate.total | 内存计算，不存储 |
| 目标企业数量 | 数据查询 | 从 `search_tasks.target_count` 读取 | MySQL `search_tasks` 表 |
| 预计候选企业 | 后端计算 | 单一主数据源与 Jina Reader 的 estimated_results 之和 | 内存计算，不存储 |
| 最终输出企业 | 后端计算 | 等于 target_count | 内存计算，不存储 |

### 数据表结构

**search_tasks 表（搜客任务表）**：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| task_id | varchar(32) | 主键，任务唯一标识 |
| bot_id | varchar(32) | 外键，关联 BOT |
| tenant_id | varchar(32) | 外键，关联租户 |
| user_query | text | 用户输入的自然语言需求（已包含前端快速标签追加的内容） |
| target_count | int | 目标企业数量，范围 20-200 |
| use_default_icp | tinyint(1) | 是否使用默认 ICP 画像 |
| status | varchar(20) | 任务状态：pending/running/completed/failed |
| strategy_id | varchar(32) | 关联的策略 ID，策略生成后填充 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

**search_strategies 表（搜客策略表）**：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| strategy_id | varchar(32) | 主键，策略唯一标识 |
| task_id | varchar(32) | 外键，关联搜客任务 |
| bot_id | varchar(32) | 外键，关联 BOT |
| summary | varchar(100) | 策略摘要说明 |
| data_sources | json | 数据源列表，包含 name/priority/estimated_results/estimated_moli_ai_points/api_calls，其中 api_calls 为根据目标数量和接口限制计算的调用次数 |
| waterfall | json | 瀑布流执行计划，包含 step/source/filters |
| cost_estimate | json | 费用预估，包含 base/premium_rate/premium/total |
| ai_model | varchar(50) | 生成策略使用的 AI 模型 |
| prompt_version | varchar(10) | 使用的 Prompt 版本 |
| created_at | datetime | 创建时间 |

**moli_ai_balances 表（租户 MoliAI点数表）**：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| tenant_id | varchar(32) | 主键，租户唯一标识 |
| balance | decimal(10,2) | 当前 MoliAI点数余额 |
| updated_at | datetime | 更新时间 |

### 费用计算逻辑

#### API 定价规则

| 数据源名称 | 单价 (MoliAI点数/次) | 接口限制 | 说明 |
|------------|-------------------|----------|------|
| 出海派-海关 | 1 | 单次调用最多返回 100 条 | 按实际调用次数计费 |
| 出海派-展会 | 1 | 单次调用最多返回 100 条 | 按实际调用次数计费 |
| Google Maps | 0 | 单次调用最多返回 20 条 | 免费 |
| Jina Reader | 0 | 无限制 | 免费 |

#### 调用次数计算逻辑

**计算原则**：根据目标企业数量和所选**单一主数据源**的 API 单次返回限制，计算需要调用的次数。Jina Reader 作为强制深度验证步骤，调用次数等于主数据源预估结果数。

**公式**：
```
主数据源调用次数 = ceil(目标企业数量 / API 单次返回限制)
Jina Reader 调用次数 = 主数据源预估结果数
```

**单一主数据源及 Jina Reader 调用次数计算**：

| 数据源 | 单次返回限制 | 调用次数计算公式 | 示例 (目标 100 家) |
|--------|--------------|------------------|-------------------|
| 出海派-海关 | 100 条 | ceil(target_count / 100) | ceil(100/100) = 1 次 |
| 出海派-展会 | 100 条 | ceil(target_count / 100) | ceil(100/100) = 1 次 |
| Google Maps | 20 条 | ceil(target_count / 20) | ceil(100/20) = 5 次 |
| Jina Reader | 1 条/次 | 等于主数据源预估结果数 | 若主数据源为海关：1 × 100 × 80% = 80 次 |

#### 费用计算流程

1. **确定目标企业数量**：从 `search_tasks.target_count` 读取用户设置的目标数量
2. **确定单一主数据源**：根据 AI 策略选择的主数据源（出海派-海关/出海派-展会/Google Maps）
3. **计算主数据源调用次数**：
   - 若主数据源为海关：调用次数 = ceil(target_count / 100)
   - 若主数据源为展会：调用次数 = ceil(target_count / 100)
   - 若主数据源为 Google Maps：调用次数 = ceil(target_count / 20)
4. **计算 Jina Reader 调用次数**：
   - Jina Reader 调用次数 = 主数据源预估结果数（= 主数据源调用次数 × 单次返回限制 × 预估返回率）
5. **计算主数据源及 Jina Reader 费用**：
   - 主数据源费用 = 主数据源调用次数 × 单价（海关/展会为 1，Google Maps 为 0）
   - Jina Reader 费用 = 0（免费）
6. **计算基础费用**：base = 主数据源费用 + Jina Reader 费用
7. **计算动态溢价**：premium = base × 0.3
8. **计算总预估费用**：total = base + premium

#### 费用计算示例

**场景**：用户目标企业数量为 100 家，AI 根据意图选择**出海派-海关**为主数据源

| 数据源 | 单价 | 单次限制 | 调用次数 | 小计 (MoliAI点数) |
|--------|------|----------|----------|----------------|
| 出海派-海关（主数据源） | 1 | 100 | 1 | 1 |
| Jina Reader（深度验证） | 0 | 1 条/次 | 80 | 0 |
| **基础费用** | - | - | - | **1** |
| **动态溢价 (30%)** | - | - | - | **1** |
| **总预估费用** | - | - | - | **2** |

**场景**：用户目标企业数量为 100 家，AI 根据意图选择**Google Maps**为主数据源

| 数据源 | 单价 | 单次限制 | 调用次数 | 小计 (MoliAI点数) |
|--------|------|----------|----------|----------------|
| Google Maps（主数据源） | 0 | 20 | 5 | 0 |
| Jina Reader（深度验证） | 0 | 1 条/次 | 90 | 0 |

#### 预估结果数计算

AI 根据所选**单一主数据源**的历史平均返回率估算 `estimated_results`：

| 数据源 | 预估返回率 | 预估结果数计算公式 |

### AI 策略生成处理流程

1. **接收请求**：服务端接收 `POST /api/v1/search/strategy/generate` 请求
2. **参数校验**：
   - 校验 `bot_id` 和 `task_id` 有效性
   - 校验任务状态是否为 `pending`
3. **数据准备**：
   - 从 `search_tasks` 表查询任务详情（user_query、target_count）
   - 从 `bot_customer_profiles` 表查询 ICP 画像数据
   - 从 `moli_ai_balances` 表查询 MoliAI点数余额
4. **AI 策略生成**：
   - 构建 Prompt，包含 ICP 画像、用户需求、目标数量
   - 调用 AI 模型（优先 Kimi k2.5，备选通义千问 qwen-max）
   - 设置 `response_format: { "type": "json_object" }`
   - 超时时间：60 秒
5. **AI 输出解析**：
   - 解析 AI 返回的 JSON 数据
   - 校验必要字段：strategy_id、summary、estimated_api_calls、waterfall_execution_plan
   - 校验 waterfall_execution_plan 中**只包含主数据源步骤**，若包含 JINA_READER_API 步骤则过滤并记录日志
   - 校验主数据源名称是否在允许列表内
6. **异常处理**：
   - 若 AI 调用超时（60s），自动重试 1 次
   - 若重试仍失败，返回错误码 504
   - 若 AI 输出格式错误，自动重试 1 次
   - 若重试仍失败，返回错误码 500
7. **系统强制追加 Jina Reader 深度验证步骤**：
   - 将 `waterfall_execution_plan` 映射为 `waterfall` 数组，step 序号从 1 开始递增
   - **自动追加 Jina Reader 步骤**：在 waterfall 末尾追加 `{"step": N+1, "source": "Jina Reader", "filters": {"target": "company_website"}}`
   - **构造 data_sources**：
     - 主数据源：name = 主数据源名称, priority = 1, estimated_results = AI 估算值, api_calls = 主数据源调用次数, estimated_moli_ai_points = 主数据源费用
     - Jina Reader：name = "Jina Reader", priority = 2, estimated_results = 主数据源预估结果数, api_calls = 主数据源预估结果数, estimated_moli_ai_points = 0
   - 标记 Jina Reader 步骤为"系统强制步骤"，前端展示时特殊标注
8. **费用计算**（详见「费用计算逻辑」）：
   - **确定单一主数据源**：从 AI 生成的策略中读取选中的主数据源
   - **计算主数据源调用次数**：
     - 若主数据源为海关：调用次数 = ceil(target_count / 100)
     - 若主数据源为展会：调用次数 = ceil(target_count / 100)
     - 若主数据源为 Google Maps：调用次数 = ceil(target_count / 20)
   - **计算 Jina Reader 调用次数**：
     - Jina Reader 调用次数 = 主数据源预估结果数（对每一家企业循环验证）
   - **计算主数据源及 Jina Reader 费用**（单价 × 调用次数）：
     - 主数据源费用 = 主数据源调用次数 × 单价（海关/展会为 1，Google Maps 为 0）
     - Jina Reader 费用 = 0（免费）
   - **计算汇总费用**（MoliAI点数必须为整数）：
     - 基础费用 base = 主数据源费用 + Jina Reader 费用
     - 动态溢价 premium = ceil(base × 0.3)
     - 总预估费用 total = base + premium
   - **计算执行后余额**：moli_ai_balance_after = moli_ai_balance - total
9. **数据存储**：
   - 生成 strategy_id
   - 插入 `search_strategies` 表（包含系统追加后的完整 data_sources、waterfall、cost_estimate）
   - 更新 `search_tasks.strategy_id` 字段
10. **返回响应**：返回策略详情和费用预估（包含 Jina Reader 系统强制步骤）

### 确认执行处理流程

1. **接收请求**：服务端接收 `POST /api/v1/search/execute` 请求
2. **参数校验**：
   - 校验 `bot_id`、`task_id`、`strategy_id` 有效性
   - 校验策略是否存在且未过期（生成后 30 分钟内有效）
3. **MoliAI点数校验**：
   - 查询 `moli_ai_balances` 表获取当前余额
   - 若余额 < cost_estimate.total，返回错误码 1002
4. **任务状态更新**：
   - 更新 `search_tasks.status` = 'running'
   - 更新 `search_tasks.updated_at` = 当前时间
5. **触发执行**：
   - 异步调用搜客执行服务
   - 传递 strategy_id 和 waterfall 配置
6. **返回响应**：返回任务状态和跳转地址

你是一个智能 API 调度网关与搜客策略规划师。你必须基于用户短期搜索意图和底层 ICP 画像，生成可直接执行的、精准匹配客户需求的搜索参数。

# Input
- 短期搜索意图：{{user_query}} （若为空，请完全依赖底层画像进行默认搜索）
- 底层 ICP 画像基准：{{icp_base_profile}}
- 目标寻客数量：{{target_count}}

# Task
请将短期意图与底层画像深度融合，制定一套瀑布流式的数据抓取策略。你需要完成：
1. **意图解析**：分析用户真实搜索意图（找什么类型的客户、什么地区、什么行业特征）
2. **画像补强**：当短期意图缺少关键信息时，用 ICP 画像补全
3. **API 编排**：根据意图匹配唯一主数据源，输出可直接执行的 API 参数
4. **参数生成**：输出可直接执行的 API 参数（禁止占位符）

# Available APIs (可用数据源)

## 1. GOOGLE_MAPS_API
- **能力**：按地理位置和实体类型寻找企业
- **限制**：单次调用返回 20 条
- **适用场景**：意图包含明确地理位置、需要找线下实体企业、需要补充地理位置相关企业

## 2. CHUHAIPAI_API（海关数据）
- **能力**：查询跨国采购记录，找有真实进出口交易的企业
- **限制**：单次最多 100 条
- **适用场景**：意图强调采购某类产品、找进口商/出口商、需要交易背书的客户

## 3. CANTON_FAIR_API（展会数据）
- **能力**：查询参展商/采购商数据
- **限制**：单次最多 100 条
- **适用场景**：意图提到展会、找近期活跃在行业展会的企业、需要高意向客户

# 意图解析与 API 选择逻辑

## Step 1: 解析用户意图类型
分析 {{user_query}}，识别以下维度（缺失的维度用 ICP 画像补全）：

| 维度 | 识别内容 | 补全来源 |
|------|----------|----------|
| 产品/行业 | 用户想找什么产品的客户 | user_query > icp_base_profile.main_categories |
| 客户类型 | 进口商/分销商/终端用户等 | user_query > icp_base_profile.customer_types |
| 地理位置 | 目标国家/地区/城市 | user_query > icp_base_profile.target_regions |
| 业务特征 | 有海关记录/参展/近期活跃等 | user_query 关键词识别 |
| 企业规模 | 大型企业/中小企业 | user_query > icp_base_profile.company_scale |

## Step 2: 确定唯一主数据源（关键决策）
**根据意图特征只选择 1 个主数据源，禁止混合调用多个接口。**

**选择 GOOGLE_MAPS_API 作为主数据源的情况：**
- 意图包含明确地理位置（如"美国加州"、"欧洲"）
- 意图强调找"当地企业"、"线下门店"
- ICP 画像中 target_regions 明确且 user_query 未指定其他区域

**选择 CHUHAIPAI_API 作为主数据源的情况：**
- 意图强调"采购"、"进口商"、"供应商"
- 意图包含具体产品词且强调贸易属性
- 需要找有真实交易记录的企业

**选择 CANTON_FAIR_API 作为主数据源的情况：**
- 意图提到"展会"、"参展商"、"广交会"
- 强调找"近期活跃"、"高意向"客户
- 需要快速获取有明确采购需求的企业

## Step 3: 确定瀑布流步骤数（逐条罗列传参）
根据所选主数据源的调用限制和 {{target_count}}，**逐条罗列每次调用的具体参数**：

### 若主数据源为 GOOGLE_MAPS_API：
- 单次返回限制 20 条，调用次数 = ceil(target_count / 20)
- **必须根据调用次数逐条生成 waterfall 步骤**，每个步骤的 parameters 应包含具体的传参（如不同的搜索关键词变体、不同的地理位置细分等），不得合并为一个步骤

### 若主数据源为 CHUHAIPAI_API：
- 单次返回限制 100 条，调用次数 = ceil(target_count / 100)
- 若 target_count ≤ 100：1 个主数据源步骤
- 若 100 < target_count ≤ 200：2 个主数据源步骤（每个步骤的时间范围或产品关键词应不同，逐条罗列传参）

### 若主数据源为 CANTON_FAIR_API：
- 单次返回限制 100 条，调用次数 = ceil(target_count / 100)
- 若 target_count ≤ 100：1 个主数据源步骤
- 若 100 < target_count ≤ 200：2 个主数据源步骤（每个步骤的 category 或展会类型应不同，逐条罗列传参）

# Execution Rules

1. **单一数据源原则**：严格根据用户意图只选择 1 个主数据源，禁止同时调用多个主数据源
2. **目标寻客数量上限**：设为 {{target_count}} 家
3. **AI 只生成主数据源策略**：AI 的 `waterfall_execution_plan` 中**只能包含主数据源步骤**，禁止包含其他非主数据源步骤
4. **步骤间数据传递**：后续步骤无需在前置步骤的 parameters 中引用前序步骤的输出域名，系统会自动将前置结果传递给后续步骤
5. **逐条罗列传参**：主数据源若需多次调用，waterfall 中必须逐条列出每次调用的具体参数，不得合并

# 生成参数规则（至关重要）

## 所有步骤通用规则
- **禁止占位符**：parameters 中的每个值必须是可直接使用的具体值。禁止使用 extracted_domains_from_step_1、ICP_Matched_HS_Code、Target Region from ICP、verified_domains_from_step_2、top_ranked_domains_from_step_3 等占位文本。
- **禁止自然语言描述**：所有参数值必须是机器可解析的具体值，不要用描述性文本替代。
- **必须基于实际内容**：所有参数值必须来源于 {{user_query}} 或 {{icp_base_profile}} 的实际内容，禁止编造。

## GOOGLE_MAPS_API 参数规则
- **query 构建逻辑**：
  1. 从 user_query 或 icp_base_profile.main_categories 提取产品/行业英文关键词
  2. 从 user_query 或 icp_base_profile.target_regions 提取地理位置
  3. 可选：从 icp_base_profile.customer_types 添加客户类型修饰词
  4. **格式**：`产品/行业关键词 + 客户类型 + in + 国家/地区`
  5. **示例**：LED display distributor in United States、Audio equipment importer in Europe
- **limit 计算**：建议设为 target_count 的 2-3 倍（因为单次返回 20 条，需要多轮调用）

## CHUHAIPAI_API 参数规则
- **product**：从 ICP 画像的 main_categories 中提取英文产品关键词。例如 ICP 标注"商用LED显示屏"，则 product 应为 "LED display" 或 "LED screen"。
- **start_time / end_time**：
  - 必须为 yyyy-MM-dd 格式的具体日期
  - 默认查询近 1 年数据
  - 示例：start_time="2025-04-16", end_time="2026-04-16"
  - 禁止使用 last_12_months、近一年 等自然语言描述
- **hs_code**：若能从 ICP 产品推断出 HS 编码则填写具体编码（如 852859），否则留空字符串 ""

## CANTON_FAIR_API 参数规则
- **category**：必须从以下分类树中选取**二级子分类名**（">" 右侧，逗号分隔）原样填写，不得使用左侧一级大类名、不得翻译或改写；若无匹配则留空字符串。
- 分类树格式：`一级大类>子分类1,子分类2,...`，每行一个大类。{{canton_fair_categories}}
