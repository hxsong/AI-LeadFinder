## 数据契约

### 前端展示字段映射

| 前端展示字段 | 后端字段名 | 类型 | 来源/存储 | 说明 |
|-------------|-----------|------|----------|------|
| 任务 ID | `task_id` | `string` | 路由参数传入 | 搜客任务唯一标识，格式：`TASK-{timestamp}-{random}` |
| BOT ID | `bot_id` | `string` | 路由参数传入 | 当前 BOT 唯一标识，用于权限校验和任务隔离 |
| 整体进度 | `overall_progress` | `number` | 接口返回 | 0-100 整数，根据各阶段进度加权计算 |
| 整体状态 | `overall_status` | `string` | 接口返回 | 枚举值：`pending`/`processing`/`completed`/`failed`，服务端根据 `search_tasks` 表 `status` 字段返回 |
| 阶段编码 | `stages[].stage` | `string` | 接口返回 | 枚举值：`P16`（候选企业搜索）/`P17`（数据清洗合并）/`P18`（企业信息补充）/`P19-P20`（ICP 匹配打分） |
| 阶段名称 | `stages[].name` | `string` | 接口返回 | 对应阶段编码的中文名称，服务端根据编码映射返回 |
| 阶段状态 | `stages[].status` | `string` | 接口返回 | 枚举值：`pending`/`in_progress`/`completed`/`failed`，对应 `search_tasks` 表 `stage_status` JSON 字段中的状态值 |
| 阶段进度 | `stages[].progress` | `number` | 接口返回 | 0-100 整数，根据阶段内子任务完成比例计算 |
| 候选企业数量 | `stages[].stats.candidates_found` | `number` | 接口返回 | API 返回候选企业总数，从 `search_task_results` 表统计记录数 |
| 合并后数量 | `stages[].stats.merged` | `number` | 接口返回 | ETL 合并后数量，根据 `company_name` 合并后的记录数 |
| 已评分数量 | `stages[].stats.scored` | `number` | 接口返回 | 评分完成数量，从 `search_task_results` 表统计 `match_score IS NOT NULL` 的记录数 |
| 达标企业数量 | `stages[].stats.above_threshold` | `number` | 接口返回 | 评分 ≥ 阈值（默认 60 分）数量，从 `search_task_results` 表统计 `match_score >= 60` 的记录数 |

### ICP 匹配评分字段

| 前端展示字段 | 后端字段名 | 类型 | 来源/存储 | 说明 |
|-------------|-----------|------|----------|------|

| 购买力评分 | `buying_power_score` | `number` | 系统生成 | AI 评分：基于企业规模、行业地位推断采购能力（0-100） |
| 采购意向评分 | `purchase_intent_score` | `number` | 系统生成 | AI 评分：基于供需互补性判断采购可能性（0-100） |
| 活跃意向评分 | `active_intent_score` | `number` | 系统生成 | AI 评分：基于近期动态判断采购紧迫性（0-100） |

### 数据来源深度说明

| 前端展示字段 | 后端字段名 | 类型 | 来源/存储 | 说明（含技术处理逻辑） |
|-------------|-----------|------|----------|------------------------|
| 整体进度 | `overall_progress` | `number` | 接口返回 | 服务端根据各阶段权重加权计算：P16 权重 25% + P17 权重 15% + P18 权重 30% + P19-P20 权重 30%。公式：`overall_progress = Σ(stage_progress[i] * stage_weight[i])` |
| 候选企业数量 | `stages[].stats.candidates_found` | `number` | 接口返回 | 服务端执行 `SELECT COUNT(*) FROM search_task_results WHERE task_id = ?` 实时统计，无缓存 |
| 合并后数量 | `stages[].stats.merged` | `number` | 接口返回 | 服务端执行 `SELECT COUNT(DISTINCT company_name) FROM search_task_results WHERE task_id = ?` 统计去重后的企业数量 |
| 已评分数量 | `stages[].stats.scored` | `number` | 接口返回 | 服务端执行 `SELECT COUNT(*) FROM search_task_results WHERE task_id = ? AND match_score IS NOT NULL` 统计已完成评分的企业数 |
| 达标企业数量 | `stages[].stats.above_threshold` | `number` | 接口返回 | 服务端执行 `SELECT COUNT(*) FROM search_task_results WHERE task_id = ? AND match_score >= 60` 统计评分 ≥ 60 分的企业数 |
| 企业定位 | `candidate_icp_profile.positioning` | `string` | 系统生成 | 服务端调用通义千问 qwen-max 模型，将 Jina Reader 抓取的官网内容作为输入，通过 Prompt 模板提取企业定位。结果存储于 `search_task_results` 表的 `candidate_icp_profile` JSON 字段 |
| 需求匹配度 | `match_score` | `number` | 系统生成 | 服务端调用通义千问 qwen-max 模型，将客户 ICP 画像与候选企业 ICP 画像进行匹配度评分。评分维度包括供需关系、品类匹配、业务模式匹配。结果存储于 `search_task_results` 表的 `match_score` 字段 |

## 页面流转

### 入口

| 来源 | 触发条件 | 携带参数 |
|------|---------|---------|
| 搜客策略预览.html | 用户点击"确认执行"按钮 | `bot_id`, `task_id` |
| 搜客需求输入.html | 进入页面时检测到已有任务在执行 | `bot_id`, `task_id` |

**入口前置条件**：
1. 搜客任务已创建且存在于 `search_tasks` 表中
2. 任务状态为 `processing` 或 `pending`
3. 当前用户有权限访问该 BOT 的任务

### 出口

| 目标 | 触发条件 | 携带数据 |
|------|---------|---------|
| 搜客结果列表.html | 所有步骤完成后自动跳转 | `bot_id`, `task_id` |

### 页面状态

| 状态 | 说明 | 进入条件 | 退出条件 |
|------|------|---------|---------|
| 初始化 | 页面加载，开始轮询 | 页面加载完成 | 首次轮询请求发出 |
| 候选企业搜索中 | P16 阶段：根据策略搜索候选企业 | `stages[0].status = in_progress` | 搜索完成或失败 |
| 数据清洗合并中 | P17 阶段：根据企业名称合并 | `stages[1].status = in_progress` | 合并完成或失败 |
| 企业信息补充中 | P18 阶段：Jina Reader 抓取官网 + AI 提取 ICP 画像 | `stages[2].status = in_progress` | 补充完成或失败 |
| ICP 匹配打分中 | P19-P20 阶段：AI 匹配度评分 | `stages[3].status = in_progress` | 评分完成或失败 |
| 执行完成 | 所有阶段执行完毕 | `overall_status = completed` | 自动跳转 |
| 执行失败 | 任一阶段执行失败 | 任一 `stages[].status = failed` | 用户点击重试或返回 |

**状态流转图**：
```
初始化 --(开始)--> 候选企业搜索中 --(完成)--> 数据清洗合并中 --(完成)--> 企业信息补充中 --(完成)--> ICP 匹配打分中 --(完成)--> 执行完成 --(自动跳转)--> 搜客结果列表.html
   │                      │                      │                      │
   └──────────────────────┴──────────────────────┴──────────────────────┘
                              (任一阶段失败)
                                      │
                                      ▼
                                执行失败 --(重试)--> 初始化
```

## 业务处理逻辑

### 场景一：候选企业搜索（P16）

**触发条件**：搜客任务启动，进入候选企业搜索阶段。

**执行逻辑**：

1. 系统接收请求：任务状态接口返回 `stages[0].status = in_progress`
2. 系统执行处理：
   - 根据策略配置，通过海关数据、展会数据、地图数据独立查询匹配的企业
   - 获取原始接口返回的结构化数据（企业名称、国家、地址、电话、网站等）
   - 每查询一批企业，前端更新进度条
   - 最终输出原始候选企业列表
3. 系统返回响应：更新阶段状态和进度

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 单个 API 调用超时 | 1 分钟超时，跳过本次调用继续执行 |
| API 调用失败 | 重试 1 次，报错后立刻重试 |
| 无候选企业 | 标记阶段完成，但 `stats.candidates_found = 0` |
| API 返回数据格式异常 | 记录错误日志，跳过该企业继续处理 |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| 初始化 | 候选企业搜索中 | `stages[0].status = in_progress` |
| 候选企业搜索中 | 数据清洗合并中 | `stages[0].status = completed` |
| 候选企业搜索中 | 执行失败 | `stages[0].status = failed` |

**输出**：原始候选企业列表，进入数据清洗合并阶段。

---

### 场景二：数据清洗合并（P17）

**触发条件**：候选企业搜索完成，进入数据清洗合并阶段。

**执行逻辑**：

1. 系统接收请求：任务状态接口返回 `stages[1].status = in_progress`
2. 系统执行处理：
   - 读取原始候选企业列表
   - 根据获取数据中的候选企业名称进行数据清洗合并
   - 标准化格式：统一地址格式、电话格式、网址格式（添加 https:// 前缀）
   - 每处理一批企业，前端更新进度条
3. 系统返回响应：更新阶段状态和进度

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 企业名称为空 | 跳过该企业，记录异常 |
| 数据格式异常 | 使用默认值或跳过 |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| 候选企业搜索中 | 数据清洗合并中 | `stages[0].status = completed` |
| 数据清洗合并中 | 企业信息补充中 | `stages[1].status = completed` |
| 数据清洗合并中 | 执行失败 | `stages[1].status = failed` |

**输出**：合并后的候选企业列表，进入企业信息补充阶段。

---

### 场景三：企业信息补充（P18）

**触发条件**：数据清洗合并完成，进入企业信息补充阶段。

**执行逻辑**：

1. 系统接收请求：任务状态接口返回 `stages[2].status = in_progress`
2. 系统执行处理：
   - 针对每个候选企业进行企业信息补充
   - 从 `structured_data` 中获取企业名称和域名信息
   - 使用企业名称 + 域名（可为空）通过 Jina Reader 抓取企业官网内容
   - AI 提取候选企业的 ICP 画像（企业定位、业务类型、主营品类、核心优势、企业规模）
   - 将原始企业名称添加到 ICP 画像中
   - 存储候选企业 ICP 画像（不包含抓取结果）
   - 每处理一批企业，前端更新进度条
3. 系统返回响应：更新阶段状态和进度

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| Jina Reader 抓取失败 | 记录失败原因，该企业标记为无官网内容，继续处理其他企业 |
| AI 结构化处理失败 | 使用原始抓取内容，标记结构化失败，继续处理 |
| 官网内容为空 | 基于已有数据（海关/展会/地图）推断结构化信息 |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| 数据清洗合并中 | 企业信息补充中 | `stages[1].status = completed` |
| 企业信息补充中 | ICP 匹配打分中 | `stages[2].status = completed` |
| 企业信息补充中 | 执行失败 | `stages[2].status = failed` |

**输出**：包含候选企业 ICP 画像的数据，进入 ICP 匹配打分阶段。

---

### 场景四：ICP 匹配打分（P19-P20）

**触发条件**：企业信息补充完成，进入 ICP 匹配打分阶段。

**执行逻辑**：

1. 系统接收请求：任务状态接口返回 `stages[3].status = in_progress`
2. 系统执行处理：
   - 根据结构化数据（候选企业的 ICP 画像）和候选企业详细数据
   - 通过 AI 与当前客户企业的 ICP 画像进行匹配度打分
   - AI 模型从四个维度（需求匹配度、购买力、采购意向、活跃意向）进行评分
   - 每家企业评分完成后前端更新列表进度
3. 系统返回响应：更新阶段状态和进度

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| AI 调用失败 | 使用规则引擎替代 AI 计算（基于关键词匹配） |
| AI 评分超出 0-100 范围 | 强制截断至 0-100 |
| 评分结果为空 | 默认评分 0，不阻断流程 |
| 候选企业信息不足 | 基于已有数据合理推断，在 ai_verdict 中说明信息不足的影响 |
| 候选企业可能是竞争对手 | buying_power_score 和 purchase_intent_score 给低分 |
| AI 调用超时 | 30 秒超时，使用规则引擎替代 |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| 企业信息补充中 | ICP 匹配打分中 | `stages[2].status = completed` |
| ICP 匹配打分中 | 执行完成 | `stages[3].status = completed` |
| ICP 匹配打分中 | 执行失败 | `stages[3].status = failed` |

**输出**：按评分排序的候选企业列表，含四维评分和 AI 综合判断。

### 场景五：处理进度实时展示

**触发条件**：页面加载后，任务执行中。

**执行逻辑**：

1. 系统接收请求：页面加载，开始轮询任务状态
2. 系统执行处理：
   - 展示四大执行步骤：
     1. 候选企业搜索（海关数据 / 展会数据 / 地图数据）
     2. 数据清洗合并（根据企业名称合并）
     3. 企业信息补充（Jina Reader 抓取官网 + AI 提取 ICP 画像）
     4. ICP 匹配打分（AI 四维评分）
   - 每步骤展示子步骤状态
   - 步骤状态切换：等待中（灰色）→ 执行中（蓝色+旋转图标）→ 已完成（绿色+勾选图标）
   - 进度条动画：平滑递增，带波浪流动效果
   - 中心动画：三层同心圆，最内层旋转，外层脉冲扩散
   - 所有步骤完成后自动跳转至搜客结果列表
3. 系统返回响应：实时更新 UI

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 轮询请求超时 | 继续下一次轮询，前端保持当前状态 |
| 网络断开 | 停止轮询，前端展示网络异常提示 |
| 用户关闭页面 | 异步任务继续在后台执行，不影响处理 |
| 跳转失败 | 前端展示手动跳转按钮 |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| 初始化 | 候选企业搜索中 | `stages[0].status = in_progress` |
| 候选企业搜索中 | 数据清洗合并中 | `stages[0].status = completed` |
| 数据清洗合并中 | 企业信息补充中 | `stages[1].status = completed` |
| 企业信息补充中 | ICP 匹配打分中 | `stages[2].status = completed` |
| ICP 匹配打分中 | 执行完成 | `stages[3].status = completed` |
| 执行完成 | 已跳转 | 自动跳转至搜客结果列表 |

**输出**：实时进度 UI，处理完成自动跳转。

## 技术处理逻辑

### 数据加工规则

| 前端字段 | 加工类型 | 技术处理逻辑 | 存储位置 |
|---------|---------|--------------|---------|
| `overall_progress` | 计算 | 加权计算：P16 权重 25% + P17 权重 15% + P18 权重 30% + P19-P20 权重 30%。公式：`overall_progress = Σ(stage_progress[i] * stage_weight[i])` | `search_tasks` 表 `stage_progress` JSON 字段 |
| `candidates_found` | 查询 | `SELECT COUNT(*) FROM search_task_results WHERE task_id = ?` | 实时查询，无缓存 |
| `merged` | 查询 | `SELECT COUNT(DISTINCT company_name) FROM search_task_results WHERE task_id = ?` | 实时查询，无缓存 |
| `scored` | 查询 | `SELECT COUNT(*) FROM search_task_results WHERE task_id = ? AND match_score IS NOT NULL` | 实时查询，无缓存 |
| `above_threshold` | 查询 | `SELECT COUNT(*) FROM search_task_results WHERE task_id = ? AND match_score >= 60` | 实时查询，无缓存 |
| `candidate_icp_profile` | AI 生成 | 调用通义千问 qwen-max 模型，输入 Jina Reader 抓取的官网内容，输出结构化 ICP 画像 | `search_task_results` 表 `candidate_icp_profile` JSON 字段 |
| `match_score` | AI 生成 | 调用通义千问 qwen-max 模型，输入客户 ICP 画像与候选企业 ICP 画像，输出四维评分 | `search_task_results` 表 `match_score` 字段 |
| `website_content` | 外部 API | 调用 Jina Reader API 抓取企业官网内容，超时 30 秒 | `search_task_results` 表 `website_content` TEXT 字段 |

### 数据表结构

**search_tasks 表（搜客任务主表）**

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `id` | `string` | 主键，任务唯一标识 |
| `bot_id` | `string` | BOT ID，外键关联 bots 表 |

### 整体流程编排

搜客任务执行采用异步队列处理，整体流程如下：

```
用户点击"确认执行"
    ↓
创建任务记录（search_tasks 表，status=pending）
    ↓
推送任务到消息队列（Redis/RabbitMQ）
    ↓
工作节点消费任务
    ↓
阶段一：候选企业搜索（P16）
    - 根据策略调用数据源 API（海关/展会/地图）
    - 获取原始接口的结构化数据
    - 写入 search_task_results
    ↓
阶段二：数据清洗合并（P17）
    - 根据企业名称进行数据清洗合并
    - 标准化格式
    - 更新 search_task_results
    ↓
阶段三：企业信息补充（P18）
    - 调用 Jina Reader 抓取企业官网内容
    - 存储 website_content 字段
    - 调用 AI 模型提取候选企业 ICP 画像
    - 更新 candidate_icp_profile 字段
    ↓
阶段四：ICP 匹配打分（P19-P20）
    - 调用 AI 模型
    - 根据候选企业 ICP 画像与客户 ICP 画像进行匹配度打分
    - 四维评分计算（match_score/buying_power_score/purchase_intent_score/active_intent_score）
    - 更新评分和 ai_verdict
    ↓
标记任务完成（status=completed）
    ↓
解冻 MoliAI点数，按实际消耗结算
```

### 并发控制

1. **单 BOT 并发限制**：
   - 同一 BOT 最多 1 个搜客任务执行中
   - 通过 Redis 分布式锁实现：`lock:search_task:{bot_id}`
   - 锁过期时间：30 分钟

2. **API 调用限流**：
   - Google Maps API：每秒 10 次
   - 出海派 API：每秒 5 次
   - Jina Reader API：每秒 20 次
   - 通过令牌桶算法实现限流

### AI 处理流程

**阶段一：候选企业搜索（P16）**
- 不涉及 AI 处理，纯数据查询

**阶段二：数据清洗合并（P17）**
- 不涉及 AI 处理，纯数据处理

**阶段三：企业信息补充（P18）- AI 提取候选企业 ICP 画像**

1. **数据准备**：
   - 从 `search_task_results` 表的 `structured_data` 字段读取企业名称和域名
   - 使用企业名称 + 域名调用 Jina Reader 抓取官网内容
   - 组装 AI 输入数据（官网文本）

2. **模型调用**：
   - 调用通义千问 qwen-max 模型
   - 超时时间：30 秒
   - 重试策略：失败立即重试 1 次

3. **结果处理**：
   - 解析 JSON 响应
   - 提取结构化字段（企业定位、业务类型、主营品类、核心优势、企业规模）
   - 将原始企业名称（从 `structured_data` 获取）添加到 `company_name` 字段
   - 更新 `search_task_results` 表的 `candidate_icp_profile` 字段

**阶段四：ICP 匹配打分（P19-P20）**

1. **数据准备**：
   - 从 `search_task_results` 表读取候选企业 ICP 画像（`candidate_icp_profile`）
   - 从 `icp_profiles` 表读取客户 ICP 画像数据
   - 从 `search_task_results` 表读取海关数据、展会数据、地图数据等详细数据（从 `structured_data` 和 `raw_data` 获取）
   - 组装 AI 输入数据（见 AI 提示词章节）

2. **模型调用**：
   - 调用通义千问 qwen-max 模型
   - 超时时间：30 秒
   - 重试策略：失败立即重试 1 次

3. **结果处理**：
   - 解析 JSON 响应
   - 提取四维评分（match_score、buying_power_score、purchase_intent_score、active_intent_score）
   - 提取 AI 综合判词（ai_verdict）
   - 更新 `search_task_results` 表的四维评分和判词字段

### 计算逻辑

**整体进度计算公式**：
```
overall_progress = (P16_progress × 0.25) + (P17_progress × 0.15) + (P18_progress × 0.30) + (P19-P20_progress × 0.30)
```

**各阶段进度计算**：
- P16 进度：`progress = (已查询企业数 / 目标企业数) × 100`，每查询 10 家企业更新一次
- P17 进度：`progress = (已处理企业数 / 原始企业数) × 100`，每处理 10 家企业更新一次
- P18 进度：`progress = (已处理企业数 / 合并后企业数) × 100`，每处理 5 家企业更新一次
- P19-P20 进度：`progress = (已评分企业数 / 去重后企业数) × 100`，每评分 5 家企业更新一次

**触发时机**：各阶段执行过程中实时计算

**存储方式**：更新 `search_tasks` 表的 `stage_progress` JSON 字段

### 数据一致性保障

1. **任务状态更新**：
   - 使用数据库事务保证 `search_tasks` 表和 `search_task_results` 表数据一致性
   - 阶段状态变更时，同时更新 `updated_at` 字段

2. **MoliAI点数结算**：
   - 任务开始时冻结预估 MoliAI点数
   - 任务完成后按实际 API 调用次数结算
   - 失败任务解冻全部冻结 MoliAI点数

## 四维评分标准

### 1. match_score（需求匹配度）0-100分

**评分逻辑**：基于供应链关系强度、品类匹配精度、业务场景契合度

| 分数段 | 判定标准 |
|--------|----------|
| 90-100 | 强上下游关系，客户产品是候选企业的核心生产要素（如原材料、关键组件），品类完全匹配，业务场景高度契合 |
| 75-89  | 明确的供需关系，客户产品对候选企业业务有重要支撑作用，品类高度相关 |
| 60-74  | 存在供需关系，但客户产品属于候选企业的次要需求或替代性供应，品类相关 |
| 40-59  | 弱供需关系，品类间接相关（如同一产业链不同环节），需进一步验证 |
| 20-39  | 几乎无直接供需关系，品类关联度低 |
| 0-19   | 无供需关系，或判定为竞争对手而非潜在客户 |

**扣分项**：
- 候选企业与客户业务高度重叠（竞争关系）：直接降至 0-20 分
- 信息不足以判断供需关系：最高不超过 40 分

---

### 2. buying_power_score（购买力评分）0-100分

**评分逻辑**：基于企业规模、行业地位、地理区域经济水平、业务稳定性

| 分数段 | 判定标准 |
|--------|----------|
| 90-100 | 行业头部企业 / 大型上市公司 / 跨国企业，采购体系成熟，预算充足 |
| 75-89  | 中大型区域龙头企业，有稳定采购需求和预算规划 |
| 60-74  | 中型企业，业务稳定，有一定采购预算但需审批流程 |
| 40-59  | 小型企业，采购预算有限，决策灵活但金额较小 |
| 20-39  | 微型/初创企业，采购力弱，付款能力存疑 |
| 0-19   | 信息严重不足无法判断，或无实际采购能力（如空壳公司、已停业） |

**参考信号**：
- 海关数据：进出口频次和金额
- 展会数据：参展规模和历史
- 地图数据：门店/办公地点数量、评分和评论数
- 官网信息：员工规模、业务覆盖范围、客户案例质量

---

### 3. purchase_intent_score（采购意向评分）0-100分

**评分逻辑**：基于候选企业对客户产品的依赖程度、采购历史、业务扩张迹象

| 分数段 | 判定标准 |
|--------|----------|
| 90-100 | 业务高度依赖客户产品（如核心原材料、关键设备），有明确采购需求且持续消耗 |
| 75-89  | 有明确采购需求，客户产品对其业务有重要支撑作用，可能存在持续采购关系 |
| 60-74  | 存在采购需求，但可能有多个供应来源或替代方案 |
| 40-59  | 潜在采购需求，需市场教育或需求激发 |
| 20-39  | 采购需求不明确，或采购周期极长 |
| 0-19   | 无采购需求（自产自用、业务不相关）或为客户竞争对手 |

**强意向信号**（加分项）：
- 海关数据显示近期有相关产品进口记录
- 展会数据显示近期有相关产品采购行为
- 官网显示业务扩张、产能提升计划
- 招聘网站显示在招聘采购/供应链相关岗位

---

### 4. active_intent_score（活跃意向评分）0-100分

**评分逻辑**：基于候选企业近期动态、市场活跃度、采购紧迫性指标

| 分数段 | 判定标准 |
|--------|----------|
| 90-100 | 强烈采购信号：近期扩产/新品发布/项目招标/大额采购计划/招聘采购岗位 |
| 75-89  | 明显活跃迹象：近期参加行业展会/网站大规模更新/新业务线启动/融资完成 |
| 60-74  | 正常运营活跃：常规业务更新/社交媒体活跃/正常市场活动参与 |
| 40-59  | 活跃度一般：网站更新不频繁/无特殊市场动作/信息更新滞后 |
| 20-39  | 活跃度低：长期无更新/信息过时/市场存在感弱 |
| 0-19   | 无活跃信号：网站无法访问/长期无动态/可能已停业 |

**数据来源优先级**：
1. 海关数据：近期交易记录（6个月内为佳）
2. 展会数据：近期参展记录（1年内为佳）
3. 地图数据：营业状态、近期评论
4. 官网内容：更新日期、新闻动态

## 综合判断规则

### ai_verdict 必须包含以下内容：

1. **匹配结论**：是否为高价值潜在客户（是/否/需进一步验证）
2. **关键匹配点**：列出 2-4 个核心匹配依据
3. **风险提示**：列出 1-3 个潜在风险或不确定性
4. **跟进建议**：针对该企业的具体跟进策略

### 评分一致性校验：

- 若 match_score < 40，则 purchase_intent_score 不应高于 50
- 若 buying_power_score < 30，active_intent_score 高于 70 时需特别说明原因
- 若判定为竞争对手，match_score 和 purchase_intent_score 均应低于 20
