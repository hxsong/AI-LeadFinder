## 数据契约

### 前端展示字段映射

| 前端展示字段 | 后端字段名 | 类型 | 来源/存储 | 说明 |
|-------------|-----------|------|----------|------|
| BOT ID | `bot_id` | `string` | 路由参数传入 | 当前 BOT 唯一标识 |
| 线索 ID | `lead_id` | `string` | 系统生成 | 线索唯一标识，格式：`LEAD-{timestamp}-{random}` |
| 企业 ID | `enterprise_id` | `string` | 接口返回 | 来源任务中的企业 ID，关联 `search_task_results` 表 |
| 企业名称 | `company_name` | `string` | 接口返回 | Apollo.io 或三方 API 返回 |
| 企业官网 | `website` | `string` | 接口返回 | Apollo.io 或搜客结果 |
| 企业简介 | `company_description` | `string` | 接口返回 | 企业业务描述 |
| 线索状态 | `status` | `string` | 接口返回 | digging / completed / failed |
| 挖掘进度 | `progress` | `number` | 接口返回 | 0-100 整数，前端展示用假进度 |
| 预计剩余时间 | `estimated_time` | `string` | 后端估算 | 固定值"约 2-3 分钟" |
| 联系人数量 | `contacts_count` | `number` | 接口返回 | Apollo.io 返回总数 |
| 联系人预览 | `contacts_preview` | `array` | 接口返回 | 前 1-3 个联系人简要信息 |
| Credits 消耗 | `credits_consumed` | `number` | 接口返回 | Apollo.io 实际消耗合计 |
| 入库时间 | `created_at` | `string` | 接口返回 | ISO 8601 时间戳 |
| 入库时间显示 | `created_at_display` | `string` | 前端计算 | 格式：YYYY-MM-DD，如"2024-01-20" |

### 数据来源深度说明

| 前端展示字段 | 后端字段名 | 类型 | 来源/存储 | 说明（含技术处理逻辑） |
|-------------|-----------|------|----------|------------------------|
| 线索状态 | `status` | `string` | 系统生成 | 服务端维护状态机：digging（挖掘中）→ completed（已完成）/ failed（失败）。状态存储于 `static_leads` 表，由 Apollo.io 挖掘任务回调更新 |
| 挖掘进度 | `progress` | `number` | 系统生成 | 前端展示用假进度，模拟加载效果，实际进度由服务端状态变更驱动 |
| 预计剩余时间 | `estimated_time` | `string` | 系统生成 | 固定值"约 2-3 分钟"，不随实际进度变化 |
| 联系人预览 | `contacts_preview` | `array` | 接口返回 | 从 `contacts` 表按 `lead_id` 查询，取前 3 条记录，字段包含 name、title |
| 入库时间显示 | `created_at_display` | `string` | 前端计算 | 服务端返回 ISO 8601 格式时间戳，前端格式化为日期显示：YYYY-MM-DD |
| Credits 消耗 | `credits_consumed` | `number` | 系统生成 | 基于 Apollo.io 实际调用计费：邮箱揭示 1 Credit/个。电话不揭示，不消耗电话 Credits。从 `credits_transactions` 表按 `lead_id` 汇总计算 |

## 页面流转

### 入口

| 来源 | 触发条件 | 携带参数 |
|------|---------|---------|
| 多BOT工作台.html | 点击侧边栏"静态线索池" | `bot_id` |
| 搜客结果列表.html | 用户点击企业"深度挖掘"按钮 | `bot_id` |

**入口前置条件**：
1. 用户已登录且租户状态正常
2. BOT 实例已创建且状态正常
3. 用户有权限访问该 BOT 的线索池

### 出口

| 目标 | 触发条件 | 携带数据 |
|------|---------|---------|
| 搜客结果列表.html | 点击"返回搜客结果"按钮 | `bot_id` |
| 静态线索详情.html | 点击已完成线索卡片 | `bot_id`, `lead_id` |

### 页面状态

| 状态 | 说明 | 进入条件 | 退出条件 |
|------|------|---------|---------|
| 加载中 | 请求发出后等待数据返回 | 页面加载或筛选条件变更 | 数据返回成功或失败 |
| 正常 | 数据加载成功且列表非空 | 列表数据返回且 items.length > 0 | 用户操作或筛选后为空 |
| 空态 | 无线索数据 | 列表数据返回且 items.length = 0 | 用户点击"去搜客结果页挖掘" |
| 导出中 | 导出请求发出等待响应 | 用户点击导出按钮 | 导出完成或失败 |
| 错误 | 数据加载失败 | 网络超时或服务异常 | 用户点击重新加载 |

**状态流转图**：

```
[加载中] --(数据返回成功且非空)--> [正常]
[加载中] --(数据返回成功但为空)--> [空态]
[加载中] --(网络超时/服务异常)--> [错误]

[正常] --(筛选后无数据)--> [空态]
[正常] --(点击导出)--> [导出中] --(完成)--> [正常]

[空态] --(点击去挖掘)--> [搜客结果列表.html]
[错误] --(点击重新加载)--> [加载中]
```

## 业务处理逻辑

### 场景一：查看线索列表

**触发条件**：用户进入静态线索池页面。

**执行逻辑**：

1. 系统接收请求：页面加载，从 URL 获取 `bot_id`
2. 系统执行处理：
   - 前端请求线索列表 API，默认参数：`status=all`, `sort=created_desc`, `page=1`, `limit=20`
   - 展示两类卡片：
     - **挖掘中状态**：卡片边框高亮（蓝色 #0369A1）+ "挖掘中"标签（带旋转动画）+ 进度条（动画效果）+ 剩余时间预估
     - **已完成状态**：展示联系人头像和姓名（姓名首字母缩写）+ Credits 消耗
   - 每张卡片核心信息：企业名称、企业简介、联系人数量/Credits 消耗、入库时间
3. 系统返回响应：返回线索列表数据，前端渲染卡片列表

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 无线索数据 | 显示空态引导文案："暂无静态线索"，提供"去搜客结果页挖掘"按钮 |
| 网络超时 | 显示错误状态，提供"重新加载"按钮 |
| 服务异常 | 显示错误状态，展示错误代码 |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| 加载中 | 正常 | 数据返回成功且 items.length > 0 |
| 加载中 | 空态 | 数据返回成功但 items.length = 0 |
| 加载中 | 错误 | 网络超时或服务异常 |

**输出**：静态线索卡片列表。

### 场景二：筛选与排序

**触发条件**：用户切换筛选条件（状态筛选按钮）或排序方式。

**执行逻辑**：

1. 系统接收请求：用户点击筛选按钮（全部/挖掘中/已完成）或排序选项
2. 系统执行处理：
   - 筛选维度：
     - 挖掘状态：`all` / `digging` / `completed` / `failed`
   - 排序方式：
     - `created_desc`：入库时间倒序（默认）
   - 筛选后列表实时刷新，显示加载状态
3. 系统返回响应：返回过滤后的线索列表

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 筛选后无数据 | 显示空态引导文案："当前筛选条件下暂无数据" |
| 排序切换 | 列表重新排序，保持当前筛选条件 |

**状态变更**：无

**输出**：按条件过滤后的线索列表。

   - 已完成时详情页展示完整企业信息和联系人矩阵
3. 系统返回响应：页面跳转至静态线索详情.html

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 线索不存在 | 提示"该线索不存在"，返回线索池列表 |
| 无权限访问 | 提示"无权访问该线索"，返回线索池列表 |

**状态变更**：无

**输出**：跳转至线索详情页面。

**边界条件**：

| 导出失败 | 提示"导出失败，请稍后重试" |
| 文件生成超时 | 提示"文件生成中，请稍后到下载中心查看" |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |

### 场景五：实时进度更新

**触发条件**：页面处于挖掘中卡片展示状态。

**执行逻辑**：

1. 系统接收请求：页面加载后，对挖掘中状态的线索启动轮询
2. 系统执行处理：
   - 前端每 5 秒轮询 `GET /api/v1/leads/{lead_id}/digging-status`
   - 更新进度条宽度（假进度动画效果）、预计剩余时间、当前阶段描述
   - 当状态变为 `completed` 或 `failed` 时，停止轮询并刷新卡片状态
3. 系统返回响应：返回最新挖掘状态和进度

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 轮询超时 | 保持当前显示，下次轮询继续 |
| 用户离开页面 | 停止轮询，挖掘任务在后台继续执行 |
| 挖掘完成 | 卡片状态切换为已完成，展示联系人预览 |
| 挖掘失败 | 卡片状态切换为失败，显示失败提示 |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| 挖掘中 | 已完成 | Apollo.io 回调通知挖掘成功 |
| 挖掘中 | 失败 | Apollo.io 回调通知挖掘失败或超时 |

**输出**：实时更新的挖掘进度展示。

### 场景六：联系人挖掘流程

**触发条件**：在搜客结果列表点击深度挖掘。

**执行逻辑**：

1. 系统接收请求：用户点击深度挖掘按钮
2. 系统前置检查：
   - 检查同一域名是否已存在：查询 `static_leads` 表 `website` 字段
   - 检查并发任务数：统计 `status='digging'` 的记录数
   - 检查 Credits 余额：查询 `tenant_credits` 表 `balance` 字段
3. 系统执行处理：
   - 创建挖掘任务，插入 `static_leads` 表，状态设为 `digging`
   - 异步调用 Apollo.io API：
     - 有域名：使用 `organization_domains` 精准锁定
     - 无域名：使用 `q_organization_name` 模糊匹配
   - 查询联系人列表（限制最多返回 20 个高价值联系人）：
     - `per_page: 20` - 单次搜索最多返回 20 个联系人
     - `page: 1` - 只取第一页
     - `person_seniorities: ["executive", "senior"]` - 只获取高管和资深人员
     - `person_titles: ["CEO", "CTO", "VP", "Director", "Manager", "Head"]` - 优先决策层职位
   - 数据过滤：仅保留有 email 的记录
   - 邮箱揭示（`reveal_personal_emails: true`）：每个有效邮箱 1 Credit
   - **不揭示电话**（`reveal_phone_number: false`）：不消耗电话 Credits
   - 获取社媒信息：LinkedIn、Twitter 等（不消耗 Credits）
   - 基础联系人数据不消耗 Credits
   - 无联系人结果：0 Credits
4. 系统返回响应：返回挖掘任务状态，自动跳转至静态线索池页面

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 同一企业域名重复入库 | 提示"该企业已在线索池中，不可重复入库" |
| 单租户同时挖掘数超过 3 个 | 提示"同时进行中的任务已达上限（3 个），请等待完成后重试" |
| Credits 余额不足 | 提示"Credits 余额不足（当前：{balance}），无法执行挖掘" |
| 单企业挖掘次数超过 5 次 | 提示"该企业已达到最大挖掘次数（5 次）" |
| 页面关闭 | 挖掘任务在后台异步执行，不中断 |
| Apollo.io 接口超时 | 1 分钟超时，记录失败状态 |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| 无 | 挖掘中 | 创建挖掘任务成功 |
| 挖掘中 | 已完成 | Apollo.io 挖掘成功，数据整理完成 |
| 挖掘中 | 失败 | Apollo.io 返回错误或超时 |

**输出**：联系人数据存储，线索卡片更新为已完成。

## 技术处理逻辑

### 数据加工规则

| 前端字段 | 加工类型 | 技术处理逻辑 | 存储位置 |
|---------|---------|--------------|---------|
| 线索列表 | 多表关联查询 | 从 `static_leads` 表按 `bot_id` + `status` 筛选，关联 `contacts` 表获取联系人预览，关联 `credits_transactions` 表计算 Credits 消耗 | `static_leads`, `contacts`, `credits_transactions` |
| 挖掘进度 | 假进度 | 前端展示用假进度动画，实际状态由服务端状态变更驱动 | 内存计算，不持久化 |
| 预计剩余时间 | 固定值 | 固定返回"约 2-3 分钟" | 内存计算，不持久化 |
| 入库时间显示 | 格式化 | 服务端返回 ISO 8601 时间戳，前端格式化为 YYYY-MM-DD 日期显示 | `static_leads.created_at` |
| Credits 消耗 | 聚合计算 | 从 `credits_transactions` 表按 `lead_id` 汇总 `amount` 字段，过滤 `type='apollo_contact_reveal'` 或 `type='apollo_phone_reveal'` | `credits_transactions` |

### 联系人挖掘流程

**触发时机**：用户在搜客结果列表点击"深度挖掘"按钮

**处理步骤**：

1. **前置检查**：
   - 检查同一域名是否已存在：查询 `static_leads` 表 `website` 字段
   - 检查并发任务数：统计 `static_leads` 表 `status='digging'` 且 `tenant_id` 匹配的记录数
   - 检查 Credits 余额：查询 `tenant_credits` 表 `balance` 字段

2. **创建挖掘任务**：
   - 插入 `static_leads` 表，状态设为 `digging`，进度设为 0
   - 异步调用 Apollo.io API 开始联系人挖掘

3. **Apollo.io 调用流程**：
   - 有域名：使用 `organization_domains` 精准锁定企业
   - 无域名：使用 `q_organization_name` 模糊匹配企业
   - 查询联系人列表（限制最多返回 20 个高价值联系人）：
     - `per_page: 20` - 单次搜索最多返回 20 个联系人
     - `page: 1` - 只取第一页
     - `person_seniorities: ["executive", "senior"]` - 只获取高管和资深人员
     - `person_titles: ["CEO", "CTO", "VP", "Director", "Manager", "Head"]` - 优先决策层职位
   - 数据过滤：仅保留有 email 的记录
   - 逐个揭示邮箱（`reveal_personal_emails: true`）：每个有效邮箱 1 Credit
   - **不揭示电话**（`reveal_phone_number: false`）：不消耗电话 Credits
   - 获取社媒信息：LinkedIn、Twitter 等（不消耗 Credits）

4. **进度更新**：
   - 接收 Apollo.io 回调，更新 `static_leads.status` 字段
   - 阶段完成时更新 `static_leads.status` 为 `completed` 或 `failed`

5. **数据存储**：
   - 联系人数据写入 `contacts` 表
   - Credits 消耗写入 `credits_transactions` 表

**并发控制**：
- 单租户同时进行中的挖掘任务上限：3 个
- 单企业最大挖掘次数：5 次

## 数据流

### 创建联系人挖掘占位记录

```
POST /api/v1/leads/static
```

**请求头**：`Authorization: Bearer {tenant_token}`
**Content-Type**：`application/json`

**请求参数**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| enterprise_id | `string` | 是 | 企业 ID |
| company_name | `string` | 是 | 企业名称 |
| website | `string` | 否 | 企业官网 |
| bot_id | `string` | 是 | BOT 实例 ID |

**响应结构**：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "lead_id": "LEAD-001",
    "status": "digging"
  }
}
```

**响应码说明**：

| code | 含义 |
|------|------|
| 200 | 创建成功 |
| 400 | 参数错误 |
| 1001 | 该企业已在线索池中，不可重复入库 |
| 1002 | Credits 余额不足 |
| 1003 | 同时进行中的任务已达上限（3 个） |
| 1004 | 该企业已达到最大挖掘次数（5 次） |
| 500 | 服务异常 |

---

### 获取静态线索列表

```
GET /api/v1/leads/static
```

**请求头**：`Authorization: Bearer {tenant_token}`

**请求参数**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| bot_id | `string` | 是 | BOT ID |
| status | `string` | 否 | all / digging / completed / failed，默认 all |
| sort | `string` | 否 | created_desc，默认 created_desc |
| page | `number` | 否 | 页码，默认 1 |
| limit | `number` | 否 | 每页数量，默认 20 |

**响应结构**：

```json
{
  "code": 200,
  "message": "操作成功",
```

## 异常兜底

| 异常场景 | 检测方式 | 后端处理 | 前端表现 |
|---------|---------|---------|---------|
| 线索已存在 | 同一域名重复入库 | 拒绝入库，返回 code 1001 | Toast「该企业已在线索池中，不可重复入库」 |
| 线索不存在 | code 404 | 返回错误 | Toast「该线索不存在」 |
| 挖掘失败 | Apollo.io 返回错误或超时 | 记录错误日志，更新 status='failed' | 卡片显示失败状态，Toast「联系人挖掘失败，请重试」 |
| Credits 余额不足 | 余额 ≤ 0 | 拒绝挖掘请求，返回 code 1002 | 按钮禁用，Toast「Credits 余额不足（当前：{balance}），无法执行挖掘」 |
| 挖掘已达上限 | 已挖掘 5 次 | 拒绝挖掘请求，返回 code 1004 | 按钮禁用，Toast「该企业已达到最大挖掘次数（5 次）」 |
| 并发挖掘超限 | 同时进行 3 个任务 | 拒绝挖掘请求，返回 code 1003 | 按钮禁用，Toast「同时进行中的任务已达上限（3 个），请等待完成后重试」 |
| 导出失败 | 后端异常 | 记录错误日志 | Toast「导出失败，请稍后重试」 |
| 无数据可导出 | 筛选条件下无数据 | 返回 code 404 | Toast「当前筛选条件下无数据可导出」 |
| 网络超时 | 请求超时 | 重试 1 次 | Toast「网络连接异常，请检查网络后重试」 |
| 服务不可用 | code 500 | 记录错误日志 | Toast「服务暂时不可用，请稍后再试」 |
| 无权限访问 | code 403 | 返回错误 | Toast「无权访问该线索」 |
| 未登录 | code 401 | 返回错误 | 跳转登录页面 |
