## 数据契约

| 前端展示字段 | 后端字段名 | 类型 | 来源/存储 | 说明 |
|-------------|-----------|------|----------|------|
| BOT ID | `bot_id` | `string` | 路由参数传入 | 当前 BOT 唯一标识 |
| 访客 ID | `visitor_log_id` | `string` | 系统生成 | 系统生成 UUID |
| IP 地址（脱敏） | `ip_masked` | `string` | GeoIP 脱敏处理 | 保留前3段，最后一段脱敏为 ** |
| 地理位置 | `geo_location` | `string` | GeoIP 定位数据 | 格式：城市, 国家/地区 |
| 热度等级 | `heat_level` | `string` | 系统映射 | very_high / high / normal |
| 规则打分 | `rule_score` | `number` | 行为打分引擎计算 | 0-100 |
| 设备类型 | `device_type` | `string` | 探针 JS 采集 UA 解析 | Desktop/Mobile/Tablet |
| 操作系统 | `os` | `string` | 探针 JS 采集 UA 解析 | Windows/macOS/iOS/Android |
| 流量来源 | `traffic_source` | `string` | 流量归因解析 | Google 搜索/直接访问等 |
| 落地页 | `landing_page` | `string` | 首次访问 URL 路径 | 页面路径，不含域名 |
| 访问次数 | `visit_count` | `number` | 同一 IP 历史访问次数 | — |
| 首次访问 | `first_visit` | `string` | 接口返回 | ISO 8601 日期 |
| 最近访问 | `last_visit` | `string` | 接口返回 | ISO 8601 日期 |
| 是否已挖掘 | `is_mined` | `boolean` | 接口返回 | 该访客是否已触发深度挖掘 |
| AI 分析是否触发 | `ai_analysis_triggered` | `boolean` | 系统判断 | 规则打分 ≥ 60 时为 true |
| 深度挖掘预估 MoliAI点数 | `estimated_moli_ai_points` | `number` | 系统计算 | IP 解析 + 邮箱揭示等预估消耗 |
| 今日访客数 | `today_count` | `number` | 接口返回 | 当日访客总数 |
| 近7日访客数 | `last_7d_count` | `number` | 接口返回 | 近7日访客总数 |
| 高热度访客数 | `high_heat_count` | `number` | 接口返回 | 热度等级为 very_high/high 的访客数 |
| 已挖掘访客数 | `mined_count` | `number` | 接口返回 | is_mined = true 的访客数 |

**数据来源深度说明**：

| 前端展示字段 | 后端字段名 | 类型 | 来源/存储 | 说明（含技术处理逻辑） |
|-------------|-----------|------|----------|------------------------|
| 热度等级 | `heat_level` | `string` | 系统生成 | 基于 `rule_score` 映射：≥80 为 very_high（极高）、60-79 为 high（较高）、<60 为 normal（一般）。映射逻辑实时计算，不存储 |
| 规则打分 | `rule_score` | `number` | 系统生成 | 基于访客行为数据实时计算：页面浏览(+5)、复制文本(+15)、文件下载(+10)、表单提交(+25)、表单弃填(+15)、点击行为(+8)、多次访问(+10)。上限 100 分封顶，取 `min(计算分, 100)` |
| AI 分析是否触发 | `ai_analysis_triggered` | `boolean` | 系统生成 | 规则判断：`rule_score >= 60`。触发后调用 LLM 生成关键动作洞察和综合研判建议 |
| IP 地址（脱敏） | `ip_masked` | `string` | 系统生成 | 原始 IP 经过脱敏处理：保留前3段（IPv4）或前4组（IPv6），剩余部分替换为 `**`。示例：`192.168.1.123` → `192.168.1.**` |
| 地理位置 | `geo_location` | `string` | 系统生成 | 通过 GeoIP 数据库（MaxMind GeoIP2）查询 IP 对应的地理位置，格式化为 "城市, 国家/地区" |
| 设备类型/操作系统 | `device_type` / `os` | `string` | 系统生成 | 解析 User-Agent 字符串，使用 `ua-parser-js` 库提取设备类型和操作系统信息 |
| 流量来源 | `traffic_source` | `string` | 系统生成 | 基于 Referer 和 UTM 参数解析：识别搜索引擎（Google/Bing/百度）、直接访问、社交媒体等来源 |

## 页面流转

### 入口

| 来源 | 触发条件 | 携带参数 |
|------|---------|---------|
| 探针部署.html | 部署成功后点击"跳转访客雷达" | `bot_id` |
| 多BOT工作台.html | 点击侧边栏"访客雷达" | `bot_id` |

**入口前置条件**：
1. 用户已登录且拥有企业管理员权限
2. 当前账号已完成探针部署（已绑定域名且探针状态为 active）
3. 若未部署探针，自动跳转至探针部署页面

### 出口

| 目标 | 触发条件 | 携带数据 |
|------|---------|---------|
| 访客行为详情.html | 用户点击访客卡片 | `bot_id`, `visitor_log_id` |
| 动态线索池.html | 深度挖掘完成后自动跳转 | `bot_id` |
| 动态线索详情.html | 深度挖掘完成后自动跳转（已完成状态） | `bot_id`, `dynamic_lead_id` |

### 页面状态

| 状态 | 说明 | 进入条件 | 退出条件 |
|------|------|---------|---------|
| 加载中 | 请求发出后等待数据返回 | 页面加载或筛选条件变更 | 数据返回成功或失败 |
| 正常 | 数据完整返回，展示访客列表 | 数据返回成功且 items 非空 | 用户操作或筛选条件变更 |
| 空态 | 筛选条件下无数据 | 数据返回成功但 items 为空 | 用户调整筛选条件或刷新 |
| 深度挖掘中 | 挖掘任务进行中 | 用户点击深度挖掘按钮并确认 | 挖掘接口返回成功或失败 |
| 有进行中任务 | 检测到进行中的挖掘任务 | 页面加载时查询到进行中任务 | 自动跳转至动态线索池 |

**状态流转图**：
```
[加载中] --(数据返回成功)--> [正常]
    │                           │
    │                           │--(筛选无数据)--> [空态]
    │                           │
    └──(数据返回失败)-----> [错误状态]

[正常] --(点击深度挖掘)--> [深度挖掘中] --(挖掘完成)--> [动态线索池]
```

## 业务处理逻辑

### 场景一：正常浏览访客列表

**触发条件**：页面正常加载，无进行中任务，探针已部署且状态正常。

**执行逻辑**：

1. 系统接收请求：页面加载，携带 `bot_id`
2. 系统执行处理：
   - 页面加载时调用 `GET /api/v1/visitors` 获取访客列表（默认参数：heat_level=all, region=all, date_range=today）
   - 展示统计卡片：今日访客数 / 近7日访客数 / 高热度访客数 / 已挖掘访客数
   - 默认显示全部地区访客，默认选择当日日期
   - 右侧显示"实时更新中"状态指示器（绿色脉冲动画）
3. 系统返回响应：返回访客列表数据

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 筛选条件下无数据 | 显示空态引导文案："当前筛选条件下没有找到访客记录。请尝试调整筛选条件，或稍后再查看。" |
| 探针未部署 | 自动跳转至探针部署页面 |
| 网络超时 | 显示错误状态，提示"网络连接异常，请检查网络后重试" |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| 加载中 | 正常 | 数据返回成功且 items 非空 |
| 加载中 | 空态 | 数据返回成功但 items 为空 |
| 加载中 | 错误状态 | 数据返回失败 |

**输出**：访客卡片列表，支持筛选和翻页。

---

### 场景二：热度筛选

**触发条件**：用户点击热度标签（全部/极高/较高/一般）。

**执行逻辑**：

1. 系统接收请求：用户选择热度标签
2. 系统执行处理：
   - 热度定义：极高热度（红色，规则打分 ≥ 80）、较高热度（橙色，规则打分 60-79）、一般热度（绿色，规则打分 < 60）
   - 用户点击热度标签，列表自动过滤为对应热度等级
   - 刷新统计卡片中的高热度访客计数
   - 触发 `GET /api/v1/visitors` 接口，携带 `heat_level` 参数
3. 系统返回响应：返回过滤后的访客列表

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 筛选后无数据 | 显示空态引导文案 |
| 切换筛选时网络异常 | 保留原数据，提示网络错误 |

**状态变更**：无（保持当前状态，仅刷新数据）

**输出**：按热度等级过滤后的访客列表。

---

### 场景三：地区筛选

**触发条件**：用户从下拉菜单选择地区。

**执行逻辑**：

1. 系统接收请求：用户选择地区
2. 系统执行处理：
   - 地区选项：全部地区(all) / 北美(NA) / 欧洲(EU) / 东南亚(SEA)
   - 基于 GeoIP 定位数据过滤
   - 筛选后刷新列表和统计卡片
   - 触发 `GET /api/v1/visitors` 接口，携带 `region` 参数
3. 系统返回响应：返回过滤后的访客列表

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 筛选后无数据 | 显示空态引导文案 |

**状态变更**：无

**输出**：按地区过滤后的访客列表。

---

### 场景四：时间筛选

**触发条件**：用户选择日期范围。

**执行逻辑**：

1. 系统接收请求：用户选择日期范围
2. 系统执行处理：
   - 快捷选项：今日(today，默认) / 近7天(7d) / 近30天(30d) / 全部(all)
   - 自定义日期范围：开始日期选择器 + 结束日期选择器，点击"应用"确认
   - 结束日期不能晚于当日（today），若选择晚于当日的日期自动截断至当日
   - 日期格式：YYYY-MM-DD
   - 切换日期后自动触发数据查询
   - 触发 `GET /api/v1/visitors` 接口，携带 `date_range` 或 `start_date`/`end_date` 参数
3. 系统返回响应：返回过滤后的访客列表

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 结束日期晚于当日 | 自动截断至当日 |
| 开始日期晚于结束日期 | 交换日期或提示错误 |
| 筛选后无数据 | 显示空态引导文案 |

**状态变更**：无

**输出**：按日期范围过滤后的访客列表。

---

### 场景五：深度挖掘访客

**触发条件**：用户点击卡片右侧"深度挖掘"按钮。

**执行逻辑**：

1. 系统接收请求：用户点击深度挖掘按钮
2. 系统执行处理：
   - 弹出深度挖掘弹窗，展示计费规则：
     - IP 企业解析 = 1 MoliAI点数
     - 每获取 1 个验证邮箱 = 1 MoliAI点数
     - 附加 LinkedIn = 0 MoliAI点数（免费）
     - 附加 Twitter/FB = 0 MoliAI点数/个（免费）
   - 若检测到 UTM 参数，显示"系统将优先尝试关联静态数据中已有的企业信息"提示
   - 检查 MoliAI点数余额，若余额不足禁用确认按钮并提示充值
   - 用户确认后，前端发送 `POST /api/v1/visitors/{visitor_log_id}/reveal`
   - 进入页面时查询任务状态，若存在进行中任务自动跳转到动态线索池
3. 系统返回响应：返回挖掘任务状态

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| MoliAI点数余额 ≤ 0 | 按钮禁用，弹窗提示充值 |
| 已挖掘的访客 | 显示"已挖掘"状态，按钮禁用，不重复触发 |
| 挖掘请求超时 | 弹窗提示重试，显示"挖掘请求超时，请稍后重试" |
| 访客不存在 | Toast 提示"该访客记录不存在或已被清除" |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| 正常 | 深度挖掘中 | 用户点击深度挖掘按钮并确认 |
| 深度挖掘中 | 正常 | 挖掘接口返回失败 |
| 深度挖掘中 | 动态线索池 | 挖掘任务创建成功，自动跳转 |

**输出**：异步执行，IP 解析 + 联系人挖掘在后台进行，完成后自动进入动态线索池。

## 技术处理逻辑

### 数据加工规则

| 前端字段 | 加工类型 | 技术处理逻辑 | 存储位置 |
|---------|---------|--------------|---------|
| 热度等级 | 计算 | 基于 `rule_score` 映射：≥80 为 very_high、60-79 为 high、<60 为 normal | 实时计算，不存储 |
| 规则打分 | 计算 | 累加各行为类型分值：页面浏览(+5)、复制文本(+15)、文件下载(+10)、表单提交(+25)、表单弃填(+15)、点击行为(+8)、多次访问(+10)。上限 100 分 | `visitor_scores` 表 |
| AI 分析触发 | 判断 | `rule_score >= 60` 时触发 AI 分析 | 实时判断 |
| IP 脱敏 | 加工 | 保留前3段，最后一段替换为 `**` | 实时加工 |
| 地理位置 | 查询 | 通过 GeoIP 数据库查询 IP 对应位置 | `geoip` 库/服务 |
| 设备/系统 | 解析 | 使用 `ua-parser-js` 解析 User-Agent | 实时解析 |
| 流量来源 | 归因 | 解析 Referer 和 UTM 参数识别来源 | 实时解析 |

### 规则打分机制

**设计原则**：只根据可明确识别的行为操作类型加分，不做内容判断。

**行为类型加分规则**：

| 行为类型 | 加分值 | 说明 |
|---------|-------|------|
| 页面浏览 | +5分 | 任何页面浏览行为 |
| 复制文本 | +15分 | 任何复制操作（体现主动分析意图） |
| 文件下载 | +10分 | 任何文件下载（体现深度兴趣） |
| 表单提交 | +25分 | 完成表单提交（有明确转化意向） |
| 表单弃填 | +15分 | 有表单交互但未提交（有需求但存在顾虑） |
| 点击行为 | +8分 | 点击关键元素（按钮、链接、CTA等，体现主动交互意向） |
| 多次访问 | +10分 | 历史访问次数 ≥ 2 |

**极限值处理**：规则打分上限 100 分封顶，取 `min(计算分, 100)`。

**热度等级映射**：

| 热度等级 | 规则打分范围 | UI 颜色 |
|---------|-------------|---------|
| 极高(very_high) | ≥ 80 分 | 红色(#DC2626) |
| 较高(high) | 60-79 分 | 橙色(#F59E0B) |
| 一般(normal) | < 60 分 | 绿色(#10B981) |

### AI 分析处理逻辑

**触发条件**：
- **触发阈值**：规则打分 ≥ 60 分
- **低于阈值**：不触发 AI 分析，不显示 AI 洞察区域
- **高于阈值**：触发 AI 生成关键动作洞察和综合研判建议

**AI 输入数据**：
```
- 访客行为序列: {behavior_events}
  - 每个行为节点包含: 时间戳、行为类型(event_type)、交互对象(target_object)、行为指标(metrics)
  - 行为类型: page_view, copy_text, file_download, click, form_submit, form_abandon
- 规则打分结果: {rule_based_score}
- 热度等级: {heat_level}
```

**AI 输出内容**：

一次性生成以下内容：

1. **单行为分析** (`ai_interpretation`): 对每个高价值行为节点生成意图解读
2. **关键动作洞察** (`ai_insights`): 识别高价值行为并解读意图（数组形式）
3. **综合研判建议** (`ai_reasoning`): 给出跟进建议（字符串）

**AI Output JSON Format**：

```json
{
  "ai_analysis_triggered": true,
  "ai_interpretations": [
    {
      "event_index": 1,
      "event_type": "copy_text",
      "summary": "正在制作内部的竞品选型对比表，处于深度评估阶段"
    }
  ],
  "ai_insights": [
    "复制技术参数文本，大概率正在制作内部的竞品选型对比表",
    "在 Contact 页面输入后放弃，说明存在明确采购需求"
  ],
  "ai_reasoning": "该访客行为逻辑极度清晰，处于深度选型与比价阶段。"
}
```

**AI 字段说明**：

| 字段名 | 类型 | 说明 |
|-------|------|------|
| `event_index` | int | 行为节点在序列中的索引 |
| `event_type` | string | 行为类型 |
| `summary` | string | 行为意图的一句话总结，15-30字 |
| `ai_insights` | array | 关键行为洞察列表 |
| `ai_reasoning` | string | 综合研判结论与行动建议 |

## 数据流

### 获取访客列表

```
GET /api/v1/visitors
```

**请求头**：`Authorization: Bearer {tenant_token}`

**请求参数**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| heat_level | `string` | 否 | 热度等级：very_high / high / normal / all（默认 all） |
| region | `string` | 否 | 地区：NA / EU / SEA / all（默认 all） |
| date_range | `string` | 否 | 日期范围：today / 7d / 30d / all（默认 today） |
| start_date | `string` | 否 | 自定义开始日期，YYYY-MM-DD |
| end_date | `string` | 否 | 自定义结束日期，YYYY-MM-DD |
| page | `number` | 否 | 页码，默认 1 |
| limit | `number` | 否 | 每页数量，默认 20 |

### 调用顺序说明

系统采用**三步提示词链式调用**架构，根据访客行为序列长度智能选择调用路径：

```
┌─────────────────────────────────────────────────────────────────────┐
│  Step 1: 判断行为序列长度                                            │
│  ├─ 短会话（单段，事件数 ≤ 阈值）→ 直接执行 Step 2A                  │
│  └─ 长会话（多段，事件数 > 阈值）→ 分段后执行 Step 2A × N → Step 2B  │
└─────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────┐
│  Step 2A: 动态行为评分（访客意图分析）【必调】                        │
│  作用：核心评分引擎，对单段行为进行B2B意向判定和评分                  │
│  输出：dynamic_behavior_score, b2b_buyer_likelihood,                 │
│        behavioral_insights, ai_reasoning                             │
└─────────────────────────────────────────────────────────────────────┘
                                    │
                    ┌───────────────┴───────────────┐
                    │                               │
                    ▼                               ▼
┌──────────────────────────────────┐  ┌──────────────────────────────────┐
│  Step 2B: 动态行为评分分段聚合     │  │  Step 3: 访客行为意图分析         │
│  【长会话时调用】                  │  │  【详情页展示时调用】              │
│  作用：合并多段分析结果为统一评估   │  │  作用：生成事件级详细洞察          │
│  输出：合并后的评分和洞察          │  │  输出：事件级insight、角色推测等   │
└──────────────────────────────────┘  └──────────────────────────────────┘
```

**调用规则**：
- **访客雷达列表页**：仅调用 Step 2A（或 Step 2A + Step 2B），获取评分和基础洞察用于卡片展示
- **访客行为详情页**：在 Step 2A（+2B）基础上，调用 Step 3 获取事件级详细洞察

### Step 2B: 动态行为评分分段聚合

**调用时机**：仅当访客行为序列被切分为多段分析后调用

**模型**：`kimi-k2.5`

**强制设置**：`response_format: json_object`, `temperature: 0.3`

**输入参数**：

| 参数名 | 类型 | 说明 |
|--------|------|------|
| `total_chunks` | number | 分段数量 |
| `chunk_results` | array | 各段行为分析结果（即Step 2A的输出数组），每段包含dynamic_behavior_score, b2b_buyer_likelihood, timestamp, behavioral_insights, session_duration |

**提示词模板**：

```
# Role
你是一位 B2B 行为分析聚合专家，负责将分段时间窗口内的行为评分结果整合为一份统一的访客意图评估。

# Input Data
- 分段数量: {{total_chunks}}
- 各段行为分析结果: {{chunk_results}}
  - 每段包含: dynamic_behavior_score, b2b_buyer_likelihood, timestamp, behavioral_insights, session_duration

# Merge Rules (严格遵循)

## 1. 时间衰减权重
各段基础权重按时间倒序递减：
- 最近一段: 权重1.0
- 第二段: 权重0.8
- 第三段: 权重0.6
- 更早段: 权重0.4

## 2. 意图类型权重 (在基础权重上叠加)
根据该段评分区间调整权重：
- 86-100分(Extremely High): ×1.5
- 71-85分(High): ×1.3
- 51-70分(Medium): ×1.0
- 31-50分(Low): ×0.7
- 0-30分(Very Low): ×0.3

## 3. 评分合并公式
final_score = max(各段分数) × 0.5 + weighted_average(各段分数, 上述复合权重) × 0.5
结果四舍五入取整，上限100

## 4. B2B买家概率判定
取所有段中加权后的最高等级，但需满足：
- 若最高分段的权重<0.5，需结合次高分段综合判断
- 若存在"评分骤降"（相邻两段差值>40分），标记为"意图波动"，取后段为准

## 5. 行为洞察合并
- 按意图类型分组：采购决策类 > 技术选型类 > 一般浏览类
- 同类型内按confidence排序，取top 3
- 跨类型最多保留8条
- 去重规则：evidence字段相似度>80%视为重复

## 6. 异常模式识别
- "高开低走"：首段高分，末段低分 → 可能遇到阻碍或找到竞品
- "低开高走"：首段低分，末段高分 → 逐步建立信任，值得跟进
- "波动型"：分数大幅波动 → 可能多人共用设备，需谨慎判断

# Output JSON Format
{
  "dynamic_behavior_score": 82,
  "b2b_buyer_likelihood": "High",
  "score_trend": "rising",
  "anomaly_flag": null,
  "behavioral_insights": [
    {
      "type": "采购决策信号",
      "evidence": "在代理政策页面停留3分20秒",
      "confidence": 0.88,
      "source_chunk": 3
    }
  ],
  "chunk_summary": [
    {"chunk_id": 1, "score": 45, "likelihood": "Medium"},
    {"chunk_id": 2, "score": 68, "likelihood": "Medium"},
    {"chunk_id": 3, "score": 85, "likelihood": "High"}
  ],
  "ai_reasoning": "访客意图呈明显上升趋势，从初步了解进展到代理政策研究，建议立即分配销售跟进"
}
```

| 输出字段 | 类型 | 映射到数据契约字段 |
|---------|------|-------------------|

### Step 3: 访客行为意图分析

**调用时机**：访客行为详情页展示时调用，用于生成事件级详细洞察

**模型**：`kimi-k2.5`

**强制设置**：`response_format: json_object`, `temperature: 0.3`

**输入参数**：

| 参数名 | 类型 | 说明 |
|--------|------|------|
| `session_info` | object | 会话概况（session_id, score, duration, start_time, end_time, page_count, bounce） |
| `events` | array | 行为事件明细（id, page_id, time, score, event_type） |
| `pages` | array | 浏览的页面记录（id, duration, start_time, end_time, score, scroll_depth） |
| `icp_profile` | object | 客户的理想买家画像 |

**提示词模板**：

```
# 角色
你是一位资深的 B2B 外贸网站访客行为分析师。

# 背景
一位访客刚刚在你的客户的网站上结束了一次访问。以下是系统自动采集的完整行为记录。

# 输入数据

## 会话概况
{{session_info}}
- session_id：会话编号
- score：系统累计互动得分
- duration：访问持续秒数
- start_time/end_time：起止时间
- page_count：浏览页面数
- bounce：是否跳出（仅访问1页）

## 行为事件明细 (已按时间排序)
{{events}}
- id：事件编号
- page_id：所属页面
- time：发生时间
- score：事件得分
- event_type：事件类型 (page_view/click/copy/download/form/hover)

## 页面浏览记录
{{pages}}
- id：页面编号
- duration：停留秒数
- scroll_depth：滚动深度百分比

## 理想买家画像
{{icp_profile}}

## 高价值事件 (必须生成详细insight)
- 复制技术参数
- 下载产品资料
- 表单提交/弃填
- 长时间悬停价格/联系元素(>3秒)
- 重复访问同一产品页

## 中价值事件 (生成简要insight)
- 点击CTA按钮
- 浏览产品详情页(停留>30秒)
- 滚动深度>70%

## 低价值事件 (可标记为低价值或省略)
- 快速浏览(<10秒)
- 无目的点击
- 导航菜单浏览

# 分析维度

## 一、行为路径还原
按时间顺序还原访客旅程，标注关键转折点。

## 二、购买意向判断
结合得分分布、页面停留、事件密度判断：
- 高意向：高价值事件≥3个，或表单提交
- 中意向：高价值事件1-2个，停留时长合理
- 低意向：无高价值事件，或快速跳出

## 三、访客角色推测 (支持多角色)
- 采购决策者：关注价格、政策、联系方式
- 技术评估人员：关注参数、规格、下载资料
- 竞品调研者：快速多页浏览，复制对比信息
- 无关访客：无有效交互

**注意**：访客可能兼具多角色特征，请给出主角色+副角色。

## 四、关键信号识别
- 正向信号：复制参数、下载资料、表单交互、重复访问
- 负向信号：快速离开、仅浏览非产品页、长时间无操作后关闭

## 五、事件级洞察生成规则

### 高价值事件insight (30-60字)
必须包含：行为描述 + 意图判断 + 业务含义
示例："第三次查看P1.9产品页并复制亮度参数，正在进行技术规格对比，处于选型评估阶段"

### 中价值事件insight (20-40字)
包含：行为描述 + 简要意图
示例："点击Get Quote按钮，表现出询价意向"

### 低价值事件处理
- 若连续多个低价值事件，可合并描述
- 或标记为："页面导航行为，价值较低"

### insight差异化解法
同一行为多次发生时，从以下角度区分：
- 次数递进："首次..."/"再次..."/"第三次..."
- 节奏变化："快速..."→"仔细..."
- 意图递进："初步了解"→"深入对比"→"决策准备"
- 上下文差异："在产品页..."→"在对比页..."

**输出字段映射**：

| 输出字段 | 类型 | 映射到数据契约字段 |

## 性能安全要求

### 性能指标

| 指标类型 | 指标要求 | 说明 |
|---------|---------|------|
| 响应时间 | 接口响应时间 < 500ms | 访客列表查询接口 |
| 并发量 | 支持 100 QPS | 单企业账号 |
| 数据实时性 | 访客数据延迟 < 5s | 从探针采集到页面展示 |
| 列表加载 | 首屏加载 < 1s | 包含 20 条访客记录 |

### 安全策略

| 安全维度 | 策略要求 | 说明 |
|---------|---------|------|
| 认证授权 | JWT Token 认证 | 所有接口需携带有效 Token |
| 数据脱敏 | IP 地址脱敏展示 | 前端不展示完整 IP |
| 访问控制 | 企业数据隔离 | 只能访问本企业访客数据 |
| 频率限制 | 接口限流 | 防止恶意刷取数据 |
