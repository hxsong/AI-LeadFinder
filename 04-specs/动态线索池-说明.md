## 数据契约

### 前端展示字段映射

| 前端展示字段 | 后端字段名 | 类型 | 来源/存储 | 说明 |
|-------------|-----------|------|----------|------|
| BOT ID | `bot_id` | `string` | 路由参数传入 | 当前 BOT 唯一标识 |
| 动态线索 ID | `dynamic_lead_id` | `string` | 系统生成 | 线索唯一标识，格式：`DYN-{timestamp}-{random}` |
| 访客记录 ID | `visitor_log_id` | `string` | 接口返回 | 关联的访客记录 ID |
| 线索状态 | `status` | `string` | 系统状态机 | Resolving / Completed / ISP |
| IP 地址（脱敏） | `ip_address` | `string` | GeoIP 脱敏处理 | 保留前3段，最后一段脱敏为 `**` |
| 地理位置 | `geo_location` | `string` | GeoIP 定位数据 | 格式：城市, 国家/地区 |
| 设备信息 | `device_info` | `string` | 探针 JS 采集 UA 解析 | Desktop / Windows 等组合 |
| 流量来源 | `traffic_source` | `string` | 流量归因数据 | Google 搜索 / 直接访问等 |
| IP 解析进度 | `progress.ip_resolution` | `string` | IRS IP 解析服务 | pending / in_progress / completed / failed |
| 企业验证进度 | `progress.enterprise_verification` | `string` | 企业验证服务 | pending / in_progress / completed / failed |
| 联系人挖掘进度 | `progress.contact_mining` | `string` | Apollo.io 服务 | pending / in_progress / completed / failed |
| 企业名称 | `enterprise_info.name` | `string` | IRS IP 解析服务返回 | — |
| 企业域名 | `enterprise_info.domain` | `string` | IRS IP 解析服务返回 | — |
| 行业类型 | `enterprise_info.industry` | `string` | IRS IP 解析服务返回 | — |
| 联系人数量 | `contacts_count` | `number` | Apollo.io 联系人挖掘结果 | — |
| 联系人预览 | `contacts_preview` | `array` | Apollo.io | 前 1-3 个联系人简要信息 |
| Credits 消耗 | `credits_consumed` | `number` | 接口返回 | Apollo.io Credits 实际消耗合计 |
| ISP 运营商名称 | `isp_name` | `string` | IRS IP 解析服务返回 | 宽带 ISP 名称 |
| 入库时间 | `created_at` | `string` | 系统生成 | ISO 8601 时间戳 |
| 入库时间显示 | `created_at_display` | `string` | 前端计算 | 格式：YYYY-MM-DD，如"2024-01-20" |

### 数据来源深度说明

| 前端展示字段 | 后端字段名 | 类型 | 来源/存储 | 说明（含技术处理逻辑） |
|-------------|-----------|------|----------|------------------------|
| 线索状态 | `status` | `string` | 系统生成 | 服务端维护状态机：Resolving（挖掘中）→ Completed（已完成）/ ISP（宽带匿名）。状态存储于 `dynamic_leads` 表，由 IRS IP 解析服务和 Apollo.io 挖掘任务回调更新 |
| IP 解析进度 | `progress.ip_resolution` | `string` | 系统生成 | IRS IP 解析服务状态：pending → in_progress → completed/failed。存储于 `dynamic_leads.progress` JSON 字段 |
| 企业验证进度 | `progress.enterprise_verification` | `string` | 系统生成 | 企业信息验证服务状态，验证企业名称、域名、行业信息有效性 |
| 联系人挖掘进度 | `progress.contact_mining` | `string` | 系统生成 | Apollo.io 联系人挖掘状态：pending → in_progress → completed/failed |
| 联系人预览 | `contacts_preview` | `array` | 接口返回 | 从 `dynamic_contacts` 表按 `dynamic_lead_id` 查询，取前 3 条记录，字段包含 name、title |
| 入库时间显示 | `created_at_display` | `string` | 前端计算 | 服务端返回 ISO 8601 格式时间戳，前端格式化为日期显示：YYYY-MM-DD，与静态线索池保持一致 |
| Credits 消耗 | `credits_consumed` | `number` | 系统生成 | 基于 Apollo.io 实际调用计费：IP 企业解析 1 Credit + 邮箱揭示 1 Credit/个。从 `credits_transactions` 表按 `dynamic_lead_id` 汇总计算 |
| ISP 运营商名称 | `isp_name` | `string` | 系统生成 | IRS IP 解析服务返回，当 IP 属于公共宽带时标记为 ISP 状态 |

## 页面流转

### 入口

| 来源 | 触发条件 | 携带参数 |
|------|---------|---------|
| 访客行为详情.html | 深度挖掘完成后自动跳转 | `bot_id`, `dynamic_lead_id` |
| 访客雷达.html | 点击进行中任务的进度卡片 | `bot_id`, `dynamic_lead_id` |
| 多BOT工作台.html | 点击侧边栏"动态线索池" | `bot_id` |

**入口前置条件**：
1. 用户已登录且租户状态正常
2. BOT 实例已创建且状态正常
3. 用户有权限访问该 BOT 的动态线索池

### 出口

| 目标 | 触发条件 | 携带数据 |
|------|---------|---------|
| 访客雷达.html | 点击"返回访客雷达"按钮 | `bot_id` |
| 动态线索详情.html | 点击已完成的线索卡片 | `bot_id`, `dynamic_lead_id` |

### 页面状态

| 状态 | 说明 | 进入条件 | 退出条件 |
|------|------|---------|---------|
| 加载中 | 页面加载/筛选变化 | 页面加载 | 数据返回 |
| 正常 | 数据返回成功且列表非空 | 列表数据返回且 items.length > 0 | 用户操作或筛选后为空 |
| 空态 | 无线索数据 | 列表数据返回且 items.length = 0 | 用户点击"前往访客雷达" |
| 进行中 | 存在 Resolving 任务 | 存在 Resolving 状态线索 | 任务完成或页面跳转 |
| ISP线索 | IP 解析为 ISP | IP 解析结果为公共宽带 | 用户操作 |
| 导出中 | 导出请求发出 | 点击导出按钮 | 导出完成或失败 |
| 错误 | 数据加载失败 | 网络超时或服务异常 | 用户点击重新加载 |

**状态流转图**：

```
[加载中] --(数据返回成功且非空)--> [正常]
[加载中] --(数据返回成功但为空)--> [空态]
[加载中] --(网络超时/服务异常)--> [错误]

[正常] --(存在Resolving任务)--> [进行中] --(任务完成)--> [正常]
[正常] --(IP解析为ISP)--> [ISP线索]
[正常] --(点击导出)--> [导出中] --(完成)--> [正常]
[正常] --(筛选后无数据)--> [空态]

[空态] --(点击去挖掘)--> [访客雷达.html]
[错误] --(点击重新加载)--> [加载中]
```

## 业务处理逻辑

### 场景一：查看进行中的挖掘任务

**触发条件**：系统检测到存在 Resolving 状态的线索挖掘任务。

**执行逻辑**：

1. 系统接收请求：页面加载，检测到 Resolving 任务
2. 系统执行处理：
   - 系统渲染进行中任务卡片，带脉冲动画和高亮边框（蓝色 #0369A1）
   - 卡片显示：IP 地址（脱敏）、访客基础信息（地区、设备、来源）、"挖掘解析中"状态标签（带旋转动画）
   - 进度动画展示：IP 企业解析状态 / 企业信息验证状态 / 联系人挖掘状态
   - 卡片边框高亮 + 背景渐变 + 脉冲动画
   - 显示已消耗 Credits 数量
3. 系统返回响应：实时更新任务进度

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 实时刷新 | 存在 Resolving 任务时，系统通过轮询（每 5 秒）或 WebSocket 推送实时更新任务进度 |
| 任务异步执行 | 前端页面关闭，挖掘任务在后台继续执行，不中断 |
| 接口超时 | 单次接口调用超时 1 分钟，跳过本次调用继续执行，不阻塞整体任务 |
| 轮询超时 | 保持当前显示，下次轮询继续 |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| 加载中 | 进行中 | 存在 Resolving 任务 |
| 进行中 | 正常 | 任务完成 |
| 进行中 | ISP线索 | IP 解析为 ISP |

**输出**：进行中任务卡片，实时反映任务进度。

### 场景二：查看历史挖掘记录 - 成功线索

**触发条件**：系统检测到挖掘任务已完成（状态为 Completed）。

**执行逻辑**：

1. 系统接收请求：页面加载，检测到 Completed 任务
2. 系统执行处理：
   - 系统渲染已完成线索卡片：企业名称、企业域名（超链接）、所在地、行业类型
   - 展示卡片核心信息：联系人数量、Credits 消耗、入库时间
   - 展示联系人预览：前 1-3 个联系人简要信息（头像、姓名、职位）
   - 点击卡片跳转至动态线索详情
3. 系统返回响应：页面渲染完成

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 动态线索池 vs 静态线索池 | 两类数据独立管理，互不干扰 |
| 无联系人结果 | 显示"未找到联系人" |

**状态变更**：无

**输出**：历史线索卡片列表，按入库时间倒序排列。

**边界条件**：

| 条件 | 系统行为 |

### 场景七：联系人挖掘流程（与静态线索池统一逻辑）

**触发条件**：在访客行为详情页点击深度挖掘。

**执行逻辑**：

1. 系统接收请求：用户点击深度挖掘按钮
2. 系统前置检查：
   - 检查 Credits 余额：查询 `tenant_credits` 表 `balance` 字段
   - 检查访客是否已挖掘：查询 `dynamic_leads` 表 `visitor_log_id` 字段
3. 系统执行处理：
   - 创建动态线索占位记录，插入 `dynamic_leads` 表，状态设为 `Resolving`，进度设为 0
   - **分支 A（存在有效 UTM）**：自动关联静态数据中已有的企业信息，跳过 IP 解析，直接进入企业验证和联系人挖掘阶段
   - **分支 B（无 UTM 标识）**：启动 IP 解析服务（IRS）进行企业识别，再进入企业验证和联系人挖掘阶段
   - 异步调用 Apollo.io API 进行联系人挖掘（限制最多返回 20 个高价值联系人）：
     - `per_page: 20` - 单次搜索最多返回 20 个联系人
     - `page: 1` - 只取第一页
     - `person_seniorities: ["executive", "senior"]` - 只获取高管和资深人员
     - `person_titles: ["CEO", "CTO", "VP", "Director", "Manager", "Head"]` - 优先决策层职位
   - 数据过滤：仅保留有 email 的记录
   - 邮箱揭示（`reveal_personal_emails: true`）：每个有效邮箱 1 Credit
   - **不揭示电话**（`reveal_phone_number: false`）：不消耗电话 Credits
   - 获取社媒信息：LinkedIn、Twitter 等（不消耗 Credits）
   - 基础联系人数据不消耗 Credits
   - 无联系人结果：仅消耗 IP 解析 1 Credit
4. 系统返回响应：返回挖掘任务状态，自动跳转至动态线索池页面

**边界条件**：

| 条件 | 系统行为 |
|------|---------|
| 同一访客重复挖掘 | 提示"该访客已挖掘，不可重复入库" |
| Credits 余额不足 | 提示"Credits 余额不足（当前：{balance}），无法执行挖掘" |
| IP 无法解析到企业 | 标记为 ISP 状态，进入动态线索池展示 ISP 卡片 |
| 企业信息验证未通过 | 标记为失败状态，提示"企业信息验证未通过" |
| 页面关闭 | 挖掘任务在后台异步执行，不中断 |
| Apollo.io 接口超时 | 1 分钟超时，记录失败状态 |

**状态变更**：

| 原状态 | 新状态 | 触发条件 |
|--------|--------|----------|
| 无 | Resolving | 创建挖掘任务成功 |
| Resolving | Completed | IP 解析成功 + 企业验证通过 + 联系人挖掘完成 |
| Resolving | ISP | IP 解析结果为公共宽带 |
| Resolving | Failed | IP 解析失败或企业验证未通过 |

**输出**：联系人数据存储，线索卡片更新为 Completed 或 ISP 状态。

## 技术处理逻辑

### 数据加工规则

| 前端字段 | 加工类型 | 技术处理逻辑 | 存储位置 |
|---------|---------|--------------|---------|
| 线索列表 | 多表关联查询 | 从 `dynamic_leads` 表按 `bot_id` + `status` 筛选，关联 `dynamic_contacts` 表获取联系人预览，关联 `credits_transactions` 表计算 Credits 消耗 | `dynamic_leads`, `dynamic_contacts`, `credits_transactions` |
| 挖掘进度 | 阶段状态 | 基于 `progress` JSON 字段中的三个阶段状态计算整体进度 | `dynamic_leads.progress` |
| 入库时间显示 | 格式化 | 服务端返回 ISO 8601 时间戳，前端格式化为 YYYY-MM-DD 日期显示 | `dynamic_leads.created_at` |
| Credits 消耗 | 聚合计算 | 从 `credits_transactions` 表按 `dynamic_lead_id` 汇总 `amount` 字段，过滤 `type='ip_resolution'` 或 `type='apollo_contact_reveal'` | `credits_transactions` |
| IP 脱敏 | 加工 | 保留前3段，剩余部分替换为 `**` | 上游已脱敏 |

### 联系人挖掘流程（与静态线索池统一）

**触发时机**：用户在访客行为详情页点击"深度挖掘"按钮

**处理步骤**：

1. **前置检查**：
   - 检查访客是否已挖掘：查询 `dynamic_leads` 表 `visitor_log_id` 字段
   - 检查 Credits 余额：查询 `tenant_credits` 表 `balance` 字段

2. **创建挖掘任务**：
   - 插入 `dynamic_leads` 表，状态设为 `Resolving`，`progress` 设为 `{ip_resolution: "pending", enterprise_verification: "pending", contact_mining: "pending"}`
   - 异步启动挖掘流程

3. **IP 解析阶段**（无 UTM 时）：
   - 调用 IRS IP 解析服务，通过 IP 反查获取企业域名/名称
   - 更新 `progress.ip_resolution` 状态
   - 若解析结果为 ISP，标记状态为 `ISP`，流程结束
   - 若解析失败，标记状态为 `Failed`，流程结束

4. **企业验证阶段**：
   - 验证企业信息有效性（名称、域名、行业）
   - 更新 `progress.enterprise_verification` 状态
   - 若验证未通过，标记状态为 `Failed`，流程结束

5. **Apollo.io 联系人挖掘阶段**：
   - 有域名：使用 `organization_domains` 精准锁定企业
   - 无域名：使用 `q_organization_name` 模糊匹配企业
   - 查询联系人列表（限制最多返回 20 个高价值联系人）：
     - `per_page: 20` - 单次搜索最多返回 20 个联系人
     - `page: 1` - 只取第一页
     - `person_seniorities: ["executive", "senior"]` - 只获取高管和资深人员
     - `person_titles: ["CEO", "CTO", "VP", "Director", "Manager", "Head"]` - 优先决策层职位
   - 数据过滤：仅保留有 email 的记录
   - 逐个揭示邮箱（`reveal_personal_emails: true`）：每个有效邮箱 1 Credit
   - **不揭示电话**（`reveal_phone_number: false`）：不消耗电话 Credits
   - 获取社媒信息：LinkedIn、Twitter 等（不消耗 Credits）

6. **进度更新**：
   - 接收各阶段回调，更新 `dynamic_leads.progress` 字段
   - 阶段完成时更新 `dynamic_leads.status` 为 `Completed`、`ISP` 或 `Failed`

7. **数据存储**：
   - 联系人数据写入 `dynamic_contacts` 表
   - Credits 消耗写入 `credits_transactions` 表

## 异常兜底

| 异常场景 | 检测方式 | 后端处理 | 前端表现 |
|---------|---------|---------|---------|
| IP 地址解析失败 | IP 解析服务返回失败 | 标记为 ISP 状态，记录失败原因 | ISP 线索卡片展示，Toast「IP 地址解析失败，无法识别企业信息」 |
| 企业信息验证未通过 | 验证服务返回失败 | 标记为失败状态，不进入 Completed | 失败提示，Toast「企业信息验证未通过，线索质量不达标」 |
| Credits 余额不足 | 联系人挖掘时余额不足 | 中断任务，记录欠费状态 | 任务中断，提示充值，Toast「Credits 余额不足，无法完成联系人挖掘」 |
| 挖掘任务意外中断 | 任务状态异常 | 自动重试机制，最多重试 3 次 | 自动重试尝试恢复，Toast「挖掘任务意外中断，正在尝试恢复」 |
| IP 无法解析到企业 | IP 解析服务无结果 | 标记为 ISP 状态 | ISP 线索展示，Toast「该 IP 无法解析到企业」 |
| 解析服务不可用 | IP 解析服务异常 | 任务暂停，进入重试队列 | 任务暂停，Toast「IP 解析服务暂时不可用，任务将自动重试」 |
| 网络请求超时 | 单次 API 调用超时 | 自动重试，记录超时日志 | 自动重试，Toast「网络请求超时，任务将自动重试，请稍候」 |
| 无数据可导出 | 筛选条件下无数据 | 返回 404 | 导出失败提示，Toast「当前筛选条件下无数据可导出，请调整筛选条件」 |
| 导出任务创建失败 | 后端异常 | 记录错误日志，返回 500 | 导出失败提示，Toast「导出任务创建失败，请稍后重试」 |
| 实时连接中断 | WebSocket 断开 | 尝试重连，降级为轮询 | 提示重连，Toast「实时连接已断开，正在尝试重新连接」 |
| 数据已过期 | 缓存过期 | 返回 304 或新数据 | 提示刷新，Toast「数据已过期，请刷新页面获取最新数据」 |
| 访客已挖掘 | 重复挖掘请求 | 拒绝请求，返回 409 | Toast「该访客已挖掘，不可重复入库」 |
| 线索不存在 | code 404 | 返回错误 | Toast「该线索不存在」 |
| 无权限访问 | code 403 | 返回错误 | Toast「无权访问该线索」 |
| 未登录 | code 401 | 返回错误 | 跳转登录页面 |
| 服务不可用 | code 500 | 记录错误日志 | Toast「服务暂时不可用，请稍后再试」 |

## 性能安全要求

### 性能指标

| 指标类型 | 指标要求 | 说明 |
|---------|---------|------|
| 响应时间 | 接口响应时间 < 500ms | 动态线索列表查询接口 |
| 并发量 | 支持 100 QPS | 单企业账号 |
| 数据实时性 | 进度更新延迟 < 5s | 从状态变更到页面展示 |
| 列表加载 | 首屏加载 < 1s | 包含 20 条线索记录 |
| 轮询频率 | 每 5 秒轮询一次 | Resolving 状态时的进度查询 |

### 安全策略

| 安全维度 | 策略要求 | 说明 |
|---------|---------|------|
| 认证授权 | JWT Token 认证 | 所有接口需携带有效 Token |
| 数据脱敏 | IP 地址脱敏展示 | 前端不展示完整 IP |
| 访问控制 | 企业数据隔离 | 只能访问本企业动态线索数据 |
| 频率限制 | 接口限流 | 防止恶意刷取数据 |
| 参数校验 | 严格校验路由参数 | 校验 `dynamic_lead_id` 格式和存在性 |
